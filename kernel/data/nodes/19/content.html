A service implies loose coupling. That is, a service must not depend on the application. A service exposes a concrete interface for interaction. The precondition for loose coupling is a stable interface. A service has versions. For example, using CSV and Versions.

The domain is a model (not the model of an entity in the database — the entity model is the schema). The domain is responsible for all the business processes. It knows nothing about how the data is stored, where it comes from, or how it is presented.

All the correct patterns boil down to the fact that there is always some mediator between the model and the view. It is the one responsible for that link. The model, meanwhile, knows nothing about either the mediator or the view. It only has handles, and it emits events. And you can pull those handles and process the stream of events in a dozen other ways. Observers, listeners, message passing, anything else.

Templates must contain no imperative logic in any form, even if it relates to the view itself. Only the code points where data will be inserted. There should not even be conditionals. And neither the ViewModel/Controller, nor much less the View itself, should work directly with input/output — that should be handled by mediator services. This is why going to the network directly from view components is wrong. Of course, this is all about the ideal, and in reality it all comes down to the same matter of project scale, and you should never engage in architecture astronautics or pour out patterns for the sake of patterns. But you should always carve out layers one way or another, at least in the most primitive form.

Bind extremely late and extremely dynamically. In the controller, sanitize at most; validation belongs in the model.
