REFACTORY — voiceover timing without an editing timeline
=========================================================

Question: refactory (vlang) has no timeline. Future voiceovers must be adjusted to the
schedule. Is there good UX for this, or is a timeline unavoidable?

VERDICT: no *editing* timeline needed — but a *derived, read-only* one is worth having.
The trap is thinking timeline = editor. Keep vlang text as the single source of truth
(the Motion Canvas / Remotion model: code-driven animation + generated timeline VIEW);
make voice the timing spine (the Descript thesis); capture timings by tapping during
playback (karaoke/Aegisub timing).

DECISIONS (2026-07-07)
----------------------
- #3 schedule strip: YES, strictly VIEW-ONLY (no drag-to-edit).
- #1 audio anchoring: YES, refined — a voiceover ATTACHES TO AN ELEMENT (a tx or v),
  not standalone: the element's schedule window drives when the audio plays, the same
  way `s` already gates a video's sound to its t-window. Element moves → voice moves.
- #2 tap-to-place: YES, refined — while the stage plays, a SELECTED voiceover is
  scheduled at the CURRENT ELAPSED time on tap (mark-to-place, writes t<elapsed> into
  the source). Not just cue-marking: the tap performs the scheduling.

THE VARIANTS (ordered by leverage)
----------------------------------
1. Voice as the master clock (Descript's thesis, how explainer-makers work)
   Record narration first; schedule visuals RELATIVE to speech instead of absolute
   seconds. In vlang: an audio element + named anchors —
       a intro.mp3 c0:14 #hook
       tx x60 y450 s60 cblack "це потяг?" t@hook+1
       vtitanic w200 c695:720 t>@hook
   t@name / t>@name resolve in resolveSchedule exactly like the existing t> cursor.
   Adjusting a voiceover shifts everything anchored to it — zero re-timing.
   Later upgrade: whisper word-timestamps → anchor to a spoken phrase (t@"не потяг").
   REFINED (decision above): audio attaches to an element (tx/v) rather than floating —
   attachment IS the anchor for the common case; #label anchors stay for cross-element
   references.

2. Tap-to-sync (karaoke/Aegisub timing; PowerPoint "record narration" family)
   Play the stage, tap a hotkey → timing captured in one listen-through; re-tap a
   region to fix it. One hotkey + the existing clock, no new UI surface.
   REFINED (decision above): tap = schedule the selected voiceover at current elapsed.

3. Derived schedule strip — timeline as VIEW, code as TRUTH (Motion Canvas, Remotion)
   items[] already carries resolved {start,end}: render a horizontal strip under the
   stage — one row per element, colored spans, playhead riding `elapsed`, click = seek().
   ~100 lines. Answers "why does the rect overlap the narration?" at a glance.
   DECISION: view-only; no drag-to-write-back.

4. Scrubbable time literals (Bret Victor, "Inventing on Principle")
   Text stays primary; time numbers become handles — click a t5:8 literal → stage seeks
   there; drag it → live preview (closed-form clock makes seek pure/cheap); playing
   highlights the currently-active source lines. Nice-to-have, later.

5. Duration-flow authoring (slides model)
   Elements declare durations/order; absolute times derive via t>. Narration segments
   give durations for free (cut c0:14 = 14s) — the existing "cut length drives t>" rule
   already almost does this. Not a complete answer; it's the substrate #1 rides on.

6. Full editing timeline — rejected. Drag-editing tracks = hit-testing, snapping,
   selection, undo — a month of UI for a personal tool, and it demotes vlang text to a
   serialization format. Everyone who kept code as truth (Motion Canvas, Remotion, GSAP)
   ships a viewer-timeline, not an editor-timeline.

BUILD ORDER (decided: start with #1 + #2)
-----------------------------------------
1. #1 audio element attached to tx/v (+ #label / t@label anchors) — the actual fix for
   "adjust voiceovers to schedule".
2. #2 tap-to-place — select a voiceover, play, tap → t<elapsed> written to source.
3. #3 schedule strip (view-only) — derived from items[], click = seek.
4. #4 scrubbable literals — later, if wanted.

Prior art referenced: Descript (voice/text as timing spine), Motion Canvas & Remotion
(code truth + generated timeline view), Aegisub/karaoke timing (tap-to-sync),
PowerPoint record-narration (durations from performance), Bret Victor (scrubbable
numbers / direct manipulation of code).
