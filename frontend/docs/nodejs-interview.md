## Theory (architecture, patterns, etc.)

### CI/CD

**Continuous Integration (CI):** При каждом коммите в репозиторий CI-сервер автоматически собирает и тестирует код. Если тесты проходят успешно, код (пул-реквест) сливается в основную ветку.

**Continuous Delivery (CD):** автоматическое разворачивание в тестовую или продакшен среду.

### 119\. У чому різниця між blue/green розгортанням і rolling розгортанням?

Основна відмінність у тому, що Blue/Green розгортання використовує два окремі середовища, тоді як Rolling розгортання використовує поступове оновлення в одному середовищі.

## Database (partition, view, settings, data types)

### 41\. Що таке ORM і для чого її використовують?

ORM (Object-Relational Mapping) — це технологія, яка дозволяє програмістам працювати з базами даних, використовуючи об'єктно-орієнтовані концепції. Замість написання SQL-запитів для маніпуляції даними, програмісти можуть використовувати об'єкти і методи на своїй мові програмування. Це спрощує роботу з базою даних і робить код більш читабельним та підтримуваним.

### 43\. Що таке транзакція?

Транзакція в контексті баз даних — це одиниця роботи, яка виконується як єдине ціле. Всі операції в межах транзакції повинні бути виконані успішно або жодна з них не повинна бути виконана. Це означає, що транзакція є атомарною (неподільною) операцією.

Транзакції дозволяють гарантувати цілісність та надійність даних незалежно від зовнішніх умов (відмови апаратного/програмного забезпечення) і забезпечують консистентність у випадку одночасного доступу з боку кількох користувачів або процесів.

### 146\. Які рівні ізоляції транзакцій бувають? Плюси і мінуси?

1.  **Read Uncommitted** — транзакція може читати непідтверджені зміни інших транзакцій. Найвища продуктивність, але може призвести до неконсистентності даних.
2.  **Read Committed** — читає лише підтверджені зміни. Попереджає читання брудних даних, але дозволяє читання неповторюваних даних.
3.  **Repeatable Read** — гарантує, що транзакція може читати ті ж самі рядки багато разів без змін. Не гарантує, що інші транзакції не додадуть/видалять рядки (фантоми).
4.  **Serializable** — найвищий рівень ізоляції: всі транзакції виконуються як одна за одною. Запобігає втраті оновлення і фантомному читанню, але призводить до більшої кількості блокувань і зниження продуктивності.

### 147\. Що таке foreign key? Яку роль він виконує?

Foreign key (іноземний ключ) — це стовпець або група стовпців, який посилається на первинний або унікальний ключ іншої таблиці. Головна роль — забезпечення цілісності даних та встановлення зв'язків між таблицями.

### 149\. Чим LEFT відрізняється від INNER?

**LEFT JOIN (LEFT OUTER JOIN):** повертає всі рядки з лівої таблиці (table1) і відповідні рядки з правої (table2). Якщо в правій немає відповідного запису — повертаються NULL для її стовпців. INNER повертає лише рядки, що збігаються в обох таблицях.

### 150\. Які переваги SQL порівняно з NoSQL?

-   Чітко визначена схема (таблиці, стовпці, типи) — цілісність і консистентність даних.
-   Моделювання складних зв'язків через первинні та зовнішні ключі.
-   Нормалізація зменшує дублювання і зберігає дані структуровано.

NoSQL краще підходить для горизонтального масштабування — великі обсяги даних і високі навантаження.

### 151\. Коли слід використовувати NoSQL замість реляційної БД?

Коли дані часто змінюються або мають різнорідну структуру, яку важко покласти на реляційну модель. NoSQL дозволяє зберігати напів- або неструктуровані дані без попереднього визначення схеми (вміст документів, користувацькі профілі, логи), і краще масштабується горизонтально під великий трафік.

### 153\. Які мінуси додавання індексів?

-   Кожен індекс займає додатковий простір на диску (помітно на великих таблицях / великій кількості індексів).
-   Індекси оновлюються при кожному insert/update/delete — ці операції стають повільнішими, особливо коли індексів багато.

### 154\. Які типи індексів існують?

-   **Primary Index** — автоматично на первинному ключі, унікальний. `PRIMARY KEY (id)`
-   **Unique Index** — забезпечує унікальність значень. `UNIQUE (email)`
-   **Non-Unique Index** — дозволяє дублювання, пришвидшує пошук. `INDEX (last_name)`
-   **Composite Index** — на кількох стовпцях. `INDEX (last_name, first_name)`
-   **Full-Text Index** — повнотекстовий пошук. `FULLTEXT (content)`
-   **Spatial Index** — геопросторові дані. `SPATIAL INDEX (location)`
-   **Partitioned Index** — розділений на партиції. `INDEX (date) PARTITION BY RANGE (date)`

Також є індекси типу btree, hash тощо.

### 155\. Що таке властивість ACID?

-   **Atomicity** — всі операції транзакції виконуються як єдине ціле: «все або нічого».
-   **Consistency** — БД переходить з одного дійсного стану в інший; після завершення дотримуються всі правила цілісності.
-   **Isolation** — одночасні транзакції не впливають одна на одну; проміжні стани невидимі для інших.
-   **Durability** — після завершення результати зберігаються назавжди, навіть у разі збою системи.

## Javascript

### 22\. Відмінність між var, let і const? Чому const, якщо змінна не змінюється?

**var** — функціональна область видимості; hoisting (піднімається декларація, не ініціалізація); дозволяє повторне оголошення.

**let** — блокова область видимості; піднімається, але не ініціалізується до досягнення декларації (TDZ); без повторного оголошення.

**const** — як let, але значення не можна перепризначити після ініціалізації.

const сигналізує іншим розробникам, що значення не зміниться, і захищає від випадкової зміни — відповідає сучасним практикам іменованих констант, що допомагає ловити помилки раніше.

### 24\. Способи оголошення функції?

Function Declaration, Function Expression, Arrow function.

### 27\. Різниця між function expression і function declaration?

-   **Hoisting:** декларації піднімаються цілком; вирази — разом з оголошенням змінної, але без ініціалізації до присвоєння.
-   **Доступність:** декларації доступні у всій області з моменту створення контексту; вирази — лише після досягнення в коді.
-   **Контекст:** декларації — коли функція має бути доступна всюди; вирази — для замикань або передачі функцій як аргументів.

### 28\. Отримати з масиву чисел інший масив лише зі значеннями понад 10?

`filter`

### 29\. Як видалити елемент масиву та об'єкта?

`splice` (для масиву).

### 30\. Для чого призначений тип void?

Унарний оператор `void` змушує вираз завжди повертати `undefined` .

### 36\. Які методи Promise API? Різниця?

-   **Promise.resolve / Promise.reject** — створюють вже виконаний/відхилений Promise.
-   **Promise.all** — чекає всі, повертає масив результатів або відхиляється першою помилкою. **Promise.race** — результат/помилка першого завершеного.
-   **Promise.allSettled** — масив результатів усіх, незалежно від статусу. **Promise.any** — перший успішний, або відхилення якщо всі відхилились.

## Node.js

### 1\. Що таке Node.js?

Asynchronous event-driven JavaScript runtime для виконання JavaScript на сервері, створений Раяном Далем. Спроєктований для побудови масштабованих мережевих застосунків.

### 2\. Основні переваги та недоліки Node.js?

**Переваги:** асинхронна модель і воркери; не потрібен окремий веб-сервер перед php-fpm; той самий JS-код можна використати на клієнті й сервері; швидкість завдяки V8 (компіляція в машинний код); сприяє фулстек-розробці.

**Недоліки:** у екосистемі багато підходів до однієї задачі — плутанина в підтримці; підтримка багатопоточності слабша/менш гнучка ніж у Java/C#; менше готових фреймворків «з коробки» (довго express.js був майже єдиним).

**Не дуже підходить для:** важких обчислень (наукові розрахунки, ML, обробка великих обсягів), застосунків з наднизькою затримкою (високочастотна торгівля → C++/Rust). Для ОС/драйверів — C, C++, Rust, Assembler. Для розподілених обчислень — Erlang. Для швидкої переключуваності на зелених потоках — Go.

### 4\. Головні компоненти Node.js?

V8; libuv (event loop, асинхронний I/O з ОС — fs, socket); core-модулі (process, os, fs, net, http); C/C++ bindings; npm.

### EVENT LOOP

Кожен повний оберт event loop — це **tick** . Фази: **timers** (setTimeout / setInterval), pending callbacks, idle/prepare, **poll** , **check** (setImmediate), close callbacks. Див. [understanding process.nextTick](https://nodejs.org/en/learn/asynchronous-work/understanding-processnexttick) .

### 5\. Як один thread обробляє багато паралельних запитів?

Завдяки асинхронній моделі сервер не чекає завершення колбека і може приймати інші вхідні запити.

### 6\. Чи можливо використовувати кілька потоків? Якими модулями?

`worker_threads` ; `cluster` створює дочірні процеси (форки), кожен запускає екземпляр сервера — розподіл навантаження між процесами.

### 7\. Node.js інтерпретує чи компілює код?

V8 спочатку інтерпретує, потім компілює (JIT). Оптимізує часто виконувані ділянки (корисний мономорфний код).

### 9\. Що таке libuv i V8?

**libuv** — асинхронне виконання викликів (I/O з ОС: fs, socket), має **thread pool** (4 потоки за замовчуванням), керує event loop. Пул потоків використовується лише fs, crypto, zlib і DNS-пошуком; решта операцій — в основному потоці. **V8** — інтерпретує й виконує JS: переводить у байт-код, JIT компілює гарячі ділянки в машинний код.

### 10\. Різниця між microtasks і macrotasks?

Мікро виконуються раніше. **Microtasks** — Promises, process.nextTick(), MutationObserver (у браузері). **Macrotasks** — setTimeout, setInterval, setImmediate, асинхронні I/O, обробники подій DOM.

### 11\. Що таке стрим (stream)?

Абстракція для роботи з даними, які надходять поступово, а не всі одразу. Корисно для великих файлів/мережевих даних — обробка по частинах, мінімум пам'яті. Пов'язане поняття — backpressure.

### 12\. Види стримів?

1.  **Readable** — можна читати.
2.  **Writable** — можна записувати.
3.  **Duplex** — читати й записувати.
4.  **Transform** — Duplex, що змінює/трансформує дані під час передачі.
5.  **PassThrough** — простий Transform, що передає дані без змін (корисно для дебагу/аналізу).

### 13\. Що таке event loop? З яких компонентів складається?

У libuv 7 фаз: timers, pending callback, idle, prepare, poll, check, close callbacks (idle та prepare у Node.js не використовуються). Кожна фаза — FIFO-черга колбеків; увійшовши у фазу, event loop виконує пов'язані операції та колбеки, поки черга не спорожніє або не досягне ліміту, потім переходить до наступної.

### 67\. Для яких задач кілька процесів/потоків?

Паралельна обробка даних; веб-сервери з високим навантаженням; Big Data (розподіл обробки); обробка зображень/відео (паралельні фрагменти).

### 68\. Різниця паралельного та асинхронного програмування (серверні застосунки)?

Асинхронне — обробка запитів без блокування основного потоку, ефективні I/O операції. Паралельне — одночасне виконання кількох потоків/процесів для підвищення продуктивності.

### 69\. Типи асинхронних операцій Node.js?

fs, net, proc management.

### 70\. Які модулі Node.js?

assert, async\_hooks, buffer, child\_process, cluster, console, crypto, diagnostics\_channel, dns, fs, http, http2, net, os, process, events, streams, path, worker\_threads, crypto, C++ addons.

### 71\. Різниця між операційними помилками та помилками програміста?

**Операційна:** програма намагається підключитися до БД, але сервер недоступний через мережу. **Помилка програміста:** поділ на нуль / логічний баг, що призводить до винятку.

### 73\. Що таке libuv? Складові.

Крос-платформна бібліотека з асинхронним API для I/O; розроблена для Node.js, але може працювати незалежно. Мета — подієвий цикл + абстрагування низькорівневих системних викликів. Складові:

1.  Full-featured event loop (epoll, kqueue, IOCP, event ports).
2.  I/O інтерфейс (fs IO, network IO, сокети, DNS resolution) — асинхронні й синхронні виклики.
3.  Threads — фонове виконання завдань.
4.  Processes — створення/керування процесами, IPC.
5.  Signals — обробка SIGINT, SIGTERM тощо.
6.  Work Queues — блокуючі операції у фоні без блокування основного потоку.

Threadpool використовується для всіх fs-операцій, а також getaddrinfo/getnameinfo. Розмір — `UV_THREADPOOL_SIZE` .

### 74\. Шаблони розподілених транзакцій?

#### 1\. Two-Phase Commit (2PC)

Всі учасники або завершують успішно, або відкочують. Фаза підготовки (координатор питає готовність) → фаза коміту.

#### 2\. Three-Phase Commit (3PC)

Розширення 2PC з додатковою фазою передкоміту для надійності.

#### 3\. Saga

Послідовність локальних транзакцій з компенсуючими транзакціями у разі невдачі.

#### 4\. Eventual Consistency

Дані зрештою консистентні; реплікація з затримкою + виправлення конфліктів.

#### 5\. Event Sourcing

Кожна зміна стану — окрема подія; поточний стан відтворюється з журналу подій.

#### 6\. BASE (Basically Available, Soft state, Eventually consistent)

Висока доступність там, де повна консистентність не критична.

### 75\. Чи можна програмно контролювати виділення/звільнення пам'яті?

```
const buffer = Buffer.alloc(1024); // виділення 1 КБ
```

### 76\. Що таке Garbage Collector?

Механізм автоматичного управління пам'яттю (JS, Java, Python …). Автоматично визначає об'єкти, що більше не використовуються, і звільняє їх.

### 77\. Що таке «витік пам'яті»? Як запобігти?

Memory leak — програма займає пам'ять, яку більше не використовує, але не звільняє для повторного використання.

### 78\. Як налагодити heap out of memory?

`process.memoryUsage();` · `node --max-old-space-size=4096 your_script.js`

### 79\. Як налаштувати кешування?

Redis або пакет `node-cache` :

```
const NodeCache = require('node-cache');
const myCache = new NodeCache({ stdTTL: 100, checkperiod: 120 });
```

### 80\. child\_process vs worker\_threads vs cluster?

-   **child\_process** — запуск зовнішніх команд та інших Node.js скриптів; розподіл завдань між процесами, робота з оболонкою.
-   **worker\_threads** — багатопоточні, CPU-інтенсивні завдання без блокування головного потоку.
-   **cluster** — масштабування серверних застосунків на багатоядерних системах.

### 81\. Різниця між ES modules і CommonJS?

-   **Синтаксис:** CJS — require / module.exports; ESM — import / export.
-   **Завантаження:** CJS синхронне; ESM асинхронне.
-   **Експорт:** CJS — об'єкт; ESM — значення безпосередньо (статичний аналіз).
-   **Браузери:** CJS потребує Browserify; ESM підтримується нативно.
-   **Node.js:** CJS нативно; ESM з 12+ (наприклад, `"type": "module"` у package.json).

### 82\. Для чого EventEmitter з node:events?

Потужний інструмент для побудови асинхронних, подієво-орієнтованих застосунків — легко створювати та обробляти події, що покращує організацію коду, масштабованість і підтримуваність.

### 83\. Скільки ядер задіяно за замовчуванням?

Одне ядро.

### 84\. Що таке middleware? Чому саме там, а не в коді сервісу?

Middleware — посередник між частинами ПЗ. У Express.js — функція з доступом до request, response і наступної middleware у циклі запит-відповідь. Використовується заради розділення обов'язків (читабельність, підтримуваність) та повторного використання коду.

### 89\. Що таке Thread Pool (Worker Pool)?

Група заздалегідь створених робочих потоків, готових виконувати завдання паралельно. Мета — підвищити продуктивність за рахунок зниження накладних витрат на створення/знищення потоків.

### 90\. SIGTERM vs SIGINT?

**SIGTERM** — сигнал м'якого завершення: процес встигає очистити ресурси й зберегти стан; керований, але може бути проігнорований, і завершення може зайняти час. **SIGINT** — переривання (Ctrl+C у терміналі): зручний для інтерактивних процесів, теж можна обробити; менш підходить для фонових/серверних процесів, теж можна проігнорувати.

### 91\. Що таке backpressure у стримах? Як боротись?

Виробник уповільнює темп передачі під швидкість споживача (затримки / обмеження швидкості). Споживач надсилає виробнику сигнали свого стану (готовий/неготовий приймати) для керування потоком.

### 92\. stream.PassThrough і pipe (pipeline)?

**PassThrough** — «труба», що передає дані від виробника до споживача без обробки (відлагодження або основа складніших стримів). **stream.pipeline** — безпечно з'єднує кілька стримів, автоматично обробляючи помилки й завершення.

### 93\. Події 'data', 'end', 'error', 'finish'?

**end** — завершення читання з Readable (даних більше не буде). **finish** — для Writable: усі дані записані.

### 96\. Чи працювали з pino?

Потужний інструмент логування, що налаштовується під потреби.

### 161\. Вбудовані засоби бінарної серіалізації (аналог JSON)?

Buffer, TypedArrays, `v8.serialize` .

### 162\. Чому Node.js не однопотоковий?

Є cluster, child\_process, worker\_threads. З libuv Node.js насправді використовує багатопотоковий підхід для оптимізації асинхронних I/O операцій.

### 163\. Чим замінити deprecated fs.exists?

`fs.access`

### 165\. Що таке MessagePort і BroadcastChannel?

**MessagePort** — частина MessageChannel API, обмін повідомленнями між потоками або воркерами й основним потоком; кожен MessagePort має два кінці — port1 і port2. **BroadcastChannel** — обмін повідомленнями між контекстами одного походження без спільного основного потоку.

Пов'язане з fs: **fs.fstat(fd)** — інфо про відкритий файл за дескриптором; **fs.stat(path)** — інфо про ціль (розіменовує символьне посилання); **fs.lstat(path)** — інфо про саме посилання.

### 167\. Як пов'язані node:async\_hooks і AsyncLocalStorage?

async\_hooks відстежує життєвий цикл асинхронних операцій; AsyncLocalStorage передає/зберігає контекст між асинхронними функціями. Node.js офіційно не рекомендує async\_hooks — сценарії трекінгу контексту краще покриває стабільний AsyncLocalStorage. За словами Тимура Шемсединова, дає відчутну просадку продуктивності.

### 168\. Чого не вистачає в ESM порівняно з CJS?

Глобального контексту (у CJS доступний `global` ; ESM має власний закритий контекст). Модульних псевдозмінних — у CJS властивості модуля можна змінювати динамічно, у ESM модуль статичний.

### 169\. Як стежити за змінами файлів/директорій? Проблеми?

`fs.watch()` та `fs.watchFile()` працюють на рівні ОС — можливі проблеми синхронізації подій, деякі зміни можуть бути втрачені; можливі повторні підписки на той самий файл.

### 170\. Що можна робити за допомогою node:vm?

Виконувати JS у ізольованому контексті й контролювати виконання — зокрема обмежувати час виконання, щоб уникнути надто довгих чи зациклених скриптів.

### 171\. Як захистити SharedArrayBuffer від запису з різних worker\_threads?

Змінювати не напряму, а через `Atomics` .

### 172\. Доведіть, що модуль при завантаженні огортається у функцію (замикання)?

Кожен модуль огортається у функцію-обгортку при завантаженні, що створює замикання — це видно, якщо вивести `module.wrapper` .

### 173\. Де використовують патерн Revealing Constructor?

У Promise конструктору передається колбек `(resolve, reject) => {}` ; у конструктор стримів передається об'єкт з налаштуваннями. Тобто коли в конструктор передають функцію, що зберігається/змінює поведінку базового класу.

### 174\. Як перевизначити write для Writable без класу-спадкоємця?

```
const { Writable } = require('stream');
const myWritable = new Writable({
  write(chunk, encoding, callback) {
    callback();
  }
});

myWritable._write = function (chunk, encoding, callback) {
  console.log(`Новий запис: ${chunk.toString()}`);
  callback();
};
```

### 175\. Чому виклики з JS до аддонів на C/C++/N-API повільні?

-   **Переклад типів** — перетворення даних між JS і C/C++ витратне, особливо великі об'єкти/складні структури.
-   **Контекстні перемикання** між JS і C/C++.
-   **Оптимізація** — C/C++ код не оптимізується так само, як JS.
-   **Некоректне використання** аддонів / неефективний дизайн.

Передача пам'яті з managed у unmanaged (особливо рядки) раніше вимагала копіювання блоками. Fast calls обмежені: не повинні виділяти на JS heap і не повинні тригерити JS execution (див. [v8-fast-api-calls.h](https://chromium.googlesource.com/v8/v8/+/refs/tags/9.6.142/include/v8-fast-api-calls.h) ).

### 176\. Навіщо WASI?

WebAssembly System Interface — стандартний інтерфейс, що дає WebAssembly безпечний і портативний доступ до ОС і сервісів (файлова система, мережа, таймери), розширюючи можливості поза межами веб-середовища.

### 177\. Для чого Error.captureStackTrace?

Контролювати, де і як захоплюється стек викликів помилки — корисно для користувацьких класів помилок і детальнішої інформації при відлагодженні.

### 178\. Deprecated API та стратегія виведення?

`new Buffer(` deprecated, `fs.exists()` , async\_hooks (не рекомендується — краще AsyncLocalStorage). Виведення через Deprecation Notice.

### 179\. Чому важливе правило eslint consistent-return (оптимізація V8)?

V8 ефективніше оптимізує функції з передбачуваною структурою: коли рушій знає, що функція завжди повертає значення (або ніколи), він краще оптимізує байт-код і виконання.

### 180\. Чому додавати префікс node при завантаженні вбудованих модулів?

`node:` явно вказує на вбудований модуль, усуваючи неоднозначність з однойменним npm-пакетом.

## NestJS

### 98\. Як реалізувати свій декоратор валідації?

Використати class-validator, або власний декоратор:

```
import { BadRequestException } from '@nestjs/common';

function IsNumber(target, propertyKey, descriptor) {
  const originalMethod = descriptor.value;
  descriptor.value = function (...args) {
    const numberArgIndex = args.findIndex(
      (arg) => typeof arg === 'number' && !isNaN(arg)
    );
    if (numberArgIndex === -1) {
      throw new BadRequestException('At least one argument must be a number');
    }
    return originalMethod.apply(this, args);
  };
}

class Example {
  @IsNumber
  addNumbers(a, b) {
    return a + b;
  }
}
```
