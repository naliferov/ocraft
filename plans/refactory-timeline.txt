vlang timeline design — clips (recommended), vs t>, vs labels
=============================================================

Parked design. Not implemented yet. This concerns the vlang language interpreted by
the `refactory` script node (id 251; body lives in Postgres node_bodies, not the repo).
("vlang" = the DSL; "refactory" = the reel-factory app node that interprets it.)

The problem
-----------
vlang schedules elements on a timeline. Today (LINEAR t>): each element chains off a
single cursor — `t>` = "start when the previous finite window ends"; absolute `t5:8`
sets a fixed window. Relative `t>` makes single-line insertion free (paste a line,
everything downstream reflows). Two needs it can't express well:

- PARALLEL — several elements shown at once can only share an ABSOLUTE window
  (`t0:5` repeated on each), which doesn't reflow on insertion and isn't a movable unit.
- GROUPING / INSERT BETWEEN — treat a set of elements as one movable "scene," and
  insert another scene between two without renumbering everything.


Three models, compared
======================
Scenario used throughout: a scene of a background + title shown together, then a second
scene, and we want to INSERT a third scene between them.

--- 1. LINEAR t> (today) ---------------------------------------------------------------
  o  x0 y0 w540 h960 b#111  t0:3      <- absolute, repeated
  tx x60 y450 s60 cwhite "hi" t0:3    <- absolute, repeated (parallel = same window)
  vtitanic c0:5 t>                    <- scene 2 chains after

  parallel:   works, but the window 0:3 is written on every element (repetition)
  insert:     to add a scene at 3s you must hand-edit the vtitanic window and everything
              after — absolute times don't reflow. Painful the moment scenes overlap.
  verdict:    great for a single relative chain; weak for groups shown together.

--- 2. CLIPS (recommended) -------------------------------------------------------------
  @c0                                  <- clip at absolute 0s; elements use LOCAL time
    o  x0 y0 w540 h960 b#111           <- local t defaults to whole clip
    tx x60 y450 s60 cwhite "hi"        <- same clip -> shown together (parallel is default)
  @c>                                  <- next clip starts when this one ends (relative)
    vtitanic c0:5

  insert a scene between:
  @c0
    o  ... ; tx ...
  @c>                                  <- NEW clip inserted here
    tx x0 y0 s40 cwhite "interlude" d2
  @c>                                  <- was scene 2, now reflows automatically
    vtitanic c0:5

  parallel:   free — "same clip" IS "at the same time" (each element 0-based in the clip)
  insert:     paste a @c block; with @c> the clip chain reflows; elements inside untouched
  verdict:    two-level timeline (arrange clips coarsely, elements locally) — how every
              DAW / video editor works. Simplest model that solves BOTH needs.

--- 3. LABELS (the powerful, over-general one) -----------------------------------------
  @bg o x0 y0 w540 h960 b#111 t0:3
  @title tx x60 y450 s60 cwhite "hi" t^@bg   <- start together with bg
  vtitanic c0:5 t>@bg                        <- after bg ends

  parallel:   t^@name (start alongside a named element)
  insert:     references are by NAME, so reordering/insertion never breaks them
  power:      arbitrary dependency GRAPH — any element can cue off any other, anywhere
  cost:       you name everything; resolver becomes topo-sort + cycle detection; overkill
              for "grouped scenes on a timeline"
  verdict:    only needed for the exotic 5% (cross-element cues). Not the daily tool.

Summary
  need                     linear t>   clips        labels
  parallel display         repeat abs  DEFAULT      t^@name
  movable scene unit        no          yes (@c…)    no (flat)
  insert between            hard        easy         trivial
  local (0-based) authoring no          yes          no
  implementation            1 pass      1 pass       topo-sort + cycles
Clips win the real cases at the lowest cost. Labels are more powerful but rarely needed.


Recommended: CLIPS
==================
A clip is a named-by-time group with a LOCAL time origin. You author a clip 0-based and
place it on the absolute timeline via its header.

Syntax
  @c<N>     clip starts at absolute N seconds
  @c>       clip starts when the previous clip ends (relative -> insertion reflows)
  @c<N> d<D>  optional explicit clip duration D; else duration = last element's local end
  elements after a @c… header belong to that clip until the next @c… (flat, no indentation)

Inside a clip, ALL element timing is LOCAL (offset by the clip's absolute start):
  t2       show from local 2s
  t2:5     local window 2–5s
  t>       chain after the previous element WITHIN the clip (a per-clip local cursor)
  (bare, no t) show for the whole clip
Motion (m…), cuts (c…), color/size all work per element as normal.

Example
  @c0 d3
    o  x0 y0 w540 h960 b#111
    tx x60 y450 s60 cwhite "привіт"      <- 0–3 abs, over the bg
  @c>
    vtitanic c695:720 mx-20              <- next clip, its own local time, panning
  @c10
    tx x40 y800 s40 cwhite "10s mark"    <- an absolute-placed clip, overlaps allowed


Implementation — engine (vlang-engine.js in node 251)
-----------------------------------------------------
Still a single forward pass — no topo-sort, no cycles.

1. exec() / the run loop: recognise a `@c…` header line. Parse its start
   (absolute N, or `>` = after previous clip end) and optional `d`. Track the "current
   clip" start-offset and a per-clip local cursor; tag each following element with its
   clip so resolveSchedule can offset it.

2. common() (t branch): unchanged in shape — `t` values are parsed as before but are now
   interpreted as LOCAL to the current clip.

3. resolveSchedule(): after resolving an element's local {start,end} (via the existing
   cursor logic, but per-clip), add the clip's absolute start offset -> absolute window.
   Clip start: absolute N, or (for @c>) the running "end of previous clip" = max local
   end of the previous clip + its start. One pass, clips in program order.

Everything else (draw, clock, absolute t5:8 for un-clipped elements) stays untouched.
Elements with no @c header keep today's behaviour (top-level linear t> chain).

Verification
------------
Run in refactory (node 251), paste sources, Enter to run:
- parallel-in-clip: `@c0` with a bg `o` + a `tx` (no t) -> both show together from 0.
- local time: element `t2` inside `@c10` -> appears at absolute 12s.
- relative chain: `@c0 d3` then `@c>` clip -> second clip starts at 3s.
- insert reflow: add a `@c>` clip between two others -> later clips shift, elements inside
  are untouched.
- absolute overlap: `@c10` while another clip still runs -> both visible (overlap allowed).
- regression: existing bare-t> demo (refactory-demo, node 261; no @c headers) unchanged.


Footnote — labels (deferred)
----------------------------
If cross-element cues ever become real (start X when an unrelated Y ends), add labels:
leading `@name` on an element, `t>@name` (after end) / `t^@name` (at start), bare `t>`
= sugar for `t>@previous`. Cost: resolver becomes memoized/topo-sort with cycle detection
(`timing cycle: a -> b -> a`). Every timeline-referenced element must have a statically
known duration (video: its finite cut; image/rect/text: an explicit window) so the graph
resolves at parse time with no metadata wait. Not needed for the grouped-scene use case
that clips already cover.
