ocraft — purpose, trajectory, and next ideas
=============================================
(analysis written 2026-06-19, after the kernel/runtime restructure)

This is the companion to the per-roadmap plans in this directory. It steps back:
what is ocraft *for*, where did the last month take it, and what to build next.


1. WHAT OCRAFT IS FOR (layered purposes — honest)
-------------------------------------------------

The purposes are nested. Each outer layer is the *reason* for the inner one.

A. FORM — a personal "node OS."
   One addressable tree (`/node/:id`) where a node is anything from a note to a
   running program. The decisive, non-obvious feature: data and behavior live in
   the SAME tree — a page can also run. Not a notes app, not a job runner; both.

B. FUNCTION — the substrate for a personal life-management system.
   The README direction is explicit and is the real "what it's for":
     - the node tree   = durable memory (notes, plans, logs)
     - the scheduler   = decides when / what
     - the MCP servers = the I/O channels (Telegram, Gmail, Calendar, infra)
     - an AI assistant = the language / judgment step
   ocraft is not an end in itself; it is the OS the author intends to run their
   life on.

C. AMBITION — prove a self-extending-system thesis.
   ocraft is "open in its data plane, closed in its behavior plane." The keystone
   (handlers-as-nodes) is the intellectual core: the moment you can define a new
   node TYPE from inside the running system, "create any node type" stops being a
   slogan and becomes a fact. This is the thesis the project is trying to earn.

D. MOTIVE — the personal, usually-unstated reasons it exists at all.
   - A learning + skill-demonstration vehicle. The breadth is deliberate:
     framework-free Node (no Express), Vue 3, AssemblyScript/WASM, p5.js + Three.js
     + Tone.js, MCP servers, Deno Deploy, Docker, generative art. The
     monetization/marketing lens and the YouTube content plan make this explicit —
     ocraft is also a portfolio and a teaching artifact that shows range.
   - A dogfooding tool for its author. The author's knowledge base now lives inside ocraft as nodes;
     those notes are the test data. Single-user + localhost is
     a feature, not an unfinished state — it keeps the design honest.
   - A canvas for creative computing. The scene2d/scene3d/generative-art thread is
     expression, not productivity: "Claude as a performer with an instrument."

The short version: a node OS (form) -> to run a life on (function) -> proving a
self-extending system (ambition) -> while being a learning/portfolio/dogfood/
creative vehicle (motive).


2. THE LAST MONTH, AND WHAT IT REVEALS
--------------------------------------

Arc of 2026-05-25 -> 2026-06-19 (from git):

  - IDENTITY crystallized:  artifacts -> nodes -> "node OS." The project found its
    name and its thesis (the docs reframe, the explicit no-handler card).
  - NODE MODEL matured:     nested nodes (parentId, tree, categories), numeric ids,
    addressability, scripts-calling-scripts + WASM, the `x` context, body sidecars.
  - MEMORY populated:       the author's knowledge base migrated in and was
    translated to English -> the durable-memory organ now EXISTS and holds ~150
    nodes.
  - INFRA professionalized: repo split -> MCP restructure -> proc/service manager
    -> Prettier/ESLint -> Docker yolo -> today's kernel/runtime split.

What it reveals: the month was about BUILDING AND NAMING THE SUBSTRATE. Three of
the four life-management pillars now have a foundation — memory (nodes, populated),
automation (scheduler/tasks/services), I/O (MCP). The fourth — the assistant's
judgment loop that ties them together — is barely started (an `ai-chat` node
exists; a headless "brain" does not).


3. THE GAP — substrate built, application layer thin
----------------------------------------------------

This is the key strategic read. ocraft has become a coherent, well-built
SUBSTRATE. But the stated PURPOSE (run my life on it) needs an APPLICATION LAYER
that mostly does not exist yet:

  - The memory is populated but not navigable — 150 notes reachable only by tree/id.
    No search, no backlinks, no graph.
  - The automation runs tasks but doesn't yet DO life-management — no daily note, no
    capture inbox, no briefing.
  - The MCP I/O exists but isn't wired into a loop that produces value.
  - The "judgment" pillar — the thing that would make it feel alive — is the least
    built of the four.

And note where the roadmap points: most items DEEPEN THE PLATFORM (sandbox,
handlers, codemirror, versioning, http3, generative art). Fewer turn it into a USED
life-management system. That is the underserved direction — and it is the actual
goal (layer B above). The substrate was the means; it is being mistaken for the end.


4. NEW IDEAS (beyond the roadmap) — ordered by direction-alignment
------------------------------------------------------------------

Theme: WIRE the substrate (memory + scheduler + MCP) into loops that produce daily
value. The parts already exist; these ideas connect them.

(1) THE DAILY BRAIN LOOP  [flagship next step]
    A scheduled morning task that reads today's calendar (gcal MCP), recent Telegram
    (telegram MCP), open/overdue items in the node tree, and yesterday's daily note,
    then uses the Agent SDK HEADLESS (never the localhost ai-chat endpoint) to write
    a "today" node: agenda + suggested actions + surfaced-relevant notes.
    Why it's #1: it activates all four pillars at once into ONE visible daily
    artifact. This is the move that makes ocraft cross from "impressive substrate"
    to "a thing I open every morning."
    Leverages: daily-report skill, scheduler, MCP servers, Agent SDK (already a dep).
    Needs: ideally the ocraft-self-MCP (planned) for read/write-as-tools; a v1 can
    use direct fs/API. Unattended runs want the headless runner from the sandbox plan.

(2) CAPTURE-TO-NODE INBOX  [close the input loop]
    Extend telegram-poll: flagged Telegram messages (or a dedicated "saved messages"
    feed) become nodes under an `inbox` category; voice notes -> transcribe skill ->
    node. Makes ocraft the place stray thoughts LAND — the precondition for it being
    your memory at all.
    Leverages: telegram MCP/poll (exists), transcribe skill (exists).

(3) SEARCH + BACKLINKS + GRAPH  [make the memory navigable]
    Full-text search over node bodies (API + a Cmd-K palette), a backlinks panel
    (who links to /node/:id — the data already exists in the wikilink-converted
    hrefs), and a graph view. Turns the populated-but-opaque node tree
    into an actual knowledge base.
    Leverages: the migrated notes, the /node/:id link convention. Shares an index
    with the self-MCP's `search_nodes`.

(4) A "TODAY" DASHBOARD SCRIPT NODE  [dogfood the differentiator]
    A script node that composes (via x.x) calendar + open items + recent notes +
    service/scheduler status into a live personal dashboard, rendered with x.ui.
    This is "a note that IS a program" used for the author's own daily benefit — and
    the single best demo of ocraft's whole thesis.
    Leverages: x.x compose, x.ui, the self-MCP/API.

(5) NODE TEMPLATES + SCAFFOLDING
    Recurring structures (daily note, project, meeting, reading-note) as template
    nodes a `create-from-template` affordance (or x.create) instantiates with
    parentId + a skeleton body. Removes friction from (1)/(2). Pairs with the
    keystone (handlers-as-nodes) and the self-MCP (create_node).

(6) MEMORY DURABILITY — git-snapshot + Markdown export
    Now that the notes are the source of truth, add a scheduled task that git-commits
    the node store (versioned backup) and an export-to-Markdown task (portability /
    escape hatch). Cheap insurance for the thing the whole system now depends on.
    Complements the node-versioning plan, but at the whole-store level.

(7) COMMAND PALETTE / QUICK-CREATE  [make it feel like an OS]
    Cmd-K to jump / search / create / run. Small, high-leverage UX that makes the
    tree usable at 150+ nodes and makes the "OS" framing real. Builds on (3).

(8) RUNTIME OBSERVABILITY IN THE EDITOR
    `list-executions` is CLI-only; surface execution history + service status + logs
    in the editor so you can SEE and trust the automation. Extends the
    process-management plan (which covers services) to tasks/executions.


5. RECOMMENDED SEQUENCE
-----------------------

The month built the substrate. The highest-leverage next theme is the APPLICATION
LAYER of life-management — and there is a natural critical path:

    ocraft-self-MCP (planned)      -> the brain's hands
      + capture-to-node (2)        -> input loop
      + search/backlinks (3)       -> navigable memory
      => THE DAILY BRAIN LOOP (1)  -> the payoff: a system you actually run on

Sequence those four and ocraft stops being "an impressive substrate" and becomes
the thing the README says it is for. The platform-deepening roadmap items (sandbox,
handlers-as-nodes, codemirror) are worthy, but they serve the THESIS and LEARNING
purposes more than the life-management one — schedule them as ENABLERS where a loop
needs them (e.g., the sandbox/headless runner is a prerequisite for the daily brain
to run unattended), not as ends in themselves.

One-line strategic note: you have spent a month perfecting the means. The next
move is to use it — the cheapest version of (1)+(2)+(3) is worth more to the stated
purpose than any single platform-deepening item.
