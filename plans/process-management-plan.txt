## What & why

`runtime/serviceManager.js` already supervises long-running SERVICES the right
way: one config file per service in `runtime/services/<id>.js` (`{cmd,args,cwd,env}`),
a per-service lock (`withLock('service-<id>')`), detached spawn into a rotating log,
and pid/status persisted in `runtime/state/services/<id>.json`. The `api`, `frontend`,
`scheduler`, and `ticker` services run under it. Three gaps remain, and they group
into one piece of work because they share the same supervisor and the same state:

  (a) COVERAGE — the four MCP servers (`telegram`, `gcal`, `gmail`, `digitalocean`,
      from `.mcp.json`) are NOT services. They start/stop only as a side effect of
      whatever launches them (Claude Code, an ad-hoc `node server.js`). They have no
      unified start/stop/status/logs.
  (b) HEALTH — `autoRestart` is read into `statusOf()` and surfaced in the status
      object, but nothing acts on it. A crashed service stays dead until a human
      notices. There is no health/liveness loop, only an on-demand `kill(pid,0)`
      probe when someone calls `getService`.
  (c) VISIBILITY — `serviceManager.js` is backend-only (exposed only through
      `bin/cli.js service …`). The editor cannot list, start, stop, or watch
      services, and shows no RAM/CPU. The old Varcraft note asked for exactly this:
      "a page for managing app processes… and query RAM and CPU."

This is foundational for the Direction (a personal life-management system): the
scheduler and a future chat-node "brain" need their own processes to stay up
without a human babysitting them, and the operator needs one screen to see it all.

## Fit with the node model / architecture

- Services stay where they are — `runtime/` is the automation layer; the kernel
  (`kernel/api.js`) is the node store. Per the README, the kernel "depends on
  nothing in runtime/". So the editor must NOT import `serviceManager.js` directly,
  and `kernel/api.js` must not import it either (that would invert the dependency).
- The clean seam: a SECOND tiny HTTP surface for ops, owned by `runtime/`, which
  imports `serviceManager.js`. The editor (one client) talks to it over `/api/...`
  the same way it talks to the node store. CONFIRM the placement (see Open
  questions) — a separate `runtime/opsServer.js` is the architecturally honest
  choice; folding routes into `kernel/api.js` is fewer moving parts but breaks the
  "kernel depends on nothing in runtime" rule.
- MCP servers as services is a natural extension: an MCP server IS a long-running
  `{cmd:'node', args:['mcp-servers/telegram-mcp/server.js']}` — exactly a service
  config. The catch is transport (below).

## The MCP-under-supervisor caveat (address head-on)

The four MCP servers speak **stdio** transport. A stdio MCP server's lifecycle is
owned by its MCP CLIENT — Claude Code spawns it, pipes JSON-RPC over stdin/stdout,
and kills it. If serviceManager *also* spawns it detached (stdio → the log file),
you get a SECOND process that has no client wired to its stdin, so it just sits
there doing nothing useful. You cannot meaningfully "supervise" a stdio server as a
shared daemon — there is no socket to health-check and no client attached.

Two honest options:

  (1) STANDALONE-ONLY supervision (cheap, limited). Let serviceManager start/stop/
      log an MCP server as a bare process for *liveness and log capture only* —
      useful to confirm it boots without crashing and to tail its stderr. Make the
      limitation explicit in the config and UI: "standalone — not connected to a
      client." This is the README's stated caveat verbatim. Good enough to answer
      "does telegram-mcp even start?" but it does not put MCP traffic under the
      supervisor.

  (2) MOVE MCP to HTTP/SSE transport (real, more work). The MCP SDK supports a
      streamable-HTTP / SSE server transport. An HTTP MCP server is a normal
      long-running service: it binds a port, so it can be health-checked (probe the
      port), supervised, auto-restarted, and shared across multiple clients. This is
      the only way MCP servers become first-class supervised services. It also
      unlocks the "ocraft self-MCP" roadmap item more cleanly (an always-on HTTP MCP
      the headless scheduler can call). Cost: each server grows an HTTP transport
      entry point, and `.mcp.json` entries change from `command/args` to a URL.

  RECOMMENDATION: do (1) now (one config per server, `mcpTransport:'stdio'`,
  flagged standalone), and treat (2) as a follow-up gated on the self-MCP work —
  don't convert all four pre-emptively. CONFIRM whether any MCP server is wanted as
  a shared HTTP service soon (digitalocean is the likeliest, since the headless
  deploy path wants it).

## Crash-restart / health (act on `autoRestart`)

`autoRestart` is already declared and surfaced; give it teeth with a supervisor
tick — a small loop, NOT a per-service watchdog process (that would just move the
"who watches the watcher" problem). Two placements:

  - HOST IT IN THE SCHEDULER DAEMON. The `scheduler` service already runs a forever
    loop (`runtime/scheduler.js` `runSchedulerLoop`). Add a `superviseServices()`
    pass alongside the job pass: each tick, `listServices()`, and for any service
    with `autoRestart && !running && status !== 'stopped'` (i.e. it crashed, was not
    deliberately stopped), call `startService(id)` and log a `--- auto-restart ---`
    line. Distinguish crash from intentional stop via the existing `status` field:
    `stopService` writes `status:'stopped'`; a crash leaves `status:'running'` with a
    dead pid, or `status:'failed'`. Only restart the crash cases. (The supervisor
    must never fight a human `service stop`.)
  - BACK OFF. Track `restartCount` + `lastRestartAt` in the service state; cap
    restarts (e.g. 5 in 10 min) then mark `status:'crash-looping'` and stop trying,
    so a binary that crashes on boot doesn't spin forever. Surface that state in the
    UI in red.

  HEALTH beyond liveness (optional, later): a config-declared `healthCheck`
  (an HTTP GET that must return 200, e.g. `http://localhost:3001/api/nodes` for
  `api`). Liveness (`kill(pid,0)`) only proves the process exists, not that it's
  serving. Start with liveness; add HTTP health per-service when a service has an
  obvious endpoint.

  CAVEAT: this couples auto-restart to the scheduler service being up. That's
  acceptable (the scheduler is the system's heartbeat anyway), but it means the
  scheduler itself can't auto-restart *itself*. On the always-on box, let the OS
  init (systemd) own just the scheduler; the scheduler owns everything else. (Ties
  into the deploy-target plan — systemd unit for the scheduler.)

## Live RAM/CPU — the metric source

`kill(pid,0)` gives liveness but no resource numbers. Per-process RAM/CPU on the
detached child pids comes from one of:

  - `pidusage` (npm) — tiny, cross-platform (reads /proc on Linux, `ps`/`wmic`
    elsewhere), returns `{cpu, memory, elapsed}` for a pid. Simplest; the README
    bias is framework-free but a focused leaf util like this is in keeping with the
    existing `sharp`/`dotenv` deps. RECOMMENDED.
  - Shell out to `ps -o %cpu,%mem,rss -p <pid>` ourselves (zero deps, the
    framework-free purist path) — works on macOS dev + Linux prod, a few lines,
    but we re-implement the cross-platform parsing `pidusage` already solved.
  - `process.resourceUsage()` / `process.memoryUsage()` — NO: those report THIS
    process, not the supervised children. Not applicable.

  CAVEAT on process groups: services are spawned `detached` (their own group) and
  some are multi-process — `frontend` is `npm run dev` → npm → vite (a child), and a
  service can fork grandchildren. The stored `rec.pid` is the GROUP LEADER, so its
  own RAM/CPU undercounts the real tree (vite's memory lives in the child). For an
  accurate per-service number, sum the process tree (walk children of the group, or
  read the group's processes). `pidusage` measures a single pid; tree-summing needs
  a `pidtree`-style child enumeration. Decide per the UI's accuracy bar — CONFIRM
  whether "leader pid only" is acceptable v1 (simpler, undercounts npm-based
  services) or tree-sum is required.

## API surface (what the editor needs)

serviceManager is backend-only today; the editor needs HTTP. Mirror the node-store
route style in `kernel/api.js` (`'METHOD /path'` keys, JSON bodies). Endpoints:

  GET    /api/services            -> listServices() + a `resources` field per
                                     service ({cpu, memMb}); the list page polls or
                                     subscribes to this.
  GET    /api/services/:id        -> getService(id) (full status)
  POST   /api/services/:id/start  -> startService(id)
  POST   /api/services/:id/stop   -> stopService(id)
  POST   /api/services/:id/restart-> restartService(id)
  GET    /api/services/:id/logs?lines=200  -> readLog(id) (text)
  POST   /api/services/:id/clear-logs      -> clearLog(id)

Live updates — two routes, pick per effort:
  (A) POLL the two GETs every 1-2s from the page (dead simple, matches today's
      poll-based editor; RECOMMENDED v1). Resource sampling happens on each GET.
  (B) PUSH over the Deno ws-exchange: the local backend publishes `service:*` and
      `metrics:*` events to `wss://stream.x8.deno.net/ws`, the editor subscribes.
      This is the roadmap's "replacing the poll-based proc UI" goal and the reason
      the WS exchange exists — but it depends on the ws-exchange integration item,
      which isn't built. Don't block this plan on it; ship (A), migrate to (B) when
      the exchange is wired. For log TAILING specifically, a streaming transport
      (SSE or ws) is much nicer than re-fetching the whole tail; (A) can use a
      cheap `?since=<offset>` to fetch only appended bytes if polling feels heavy.

Auth/exposure: same posture as the rest of `/api` — localhost single-user only. But
note these endpoints can START/STOP processes, which is more dangerous than node
CRUD; keep them off any network unconditionally and document it next to the
ai-chat warning.

## Implementation plan (phased)

  Phase 1 — MCP coverage (standalone). Add `runtime/services/mcp-telegram.js`,
    `mcp-gcal.js`, `mcp-gmail.js`, `mcp-digitalocean.js` (each `{cmd:'node',
    args:['mcp-servers/<x>/server.js'], mcpTransport:'stdio'}`). Surface a
    `standalone` flag in `statusOf` so the CLI/UI can label them "boot-check only,
    not client-connected." No serviceManager logic change needed — they're configs.
  Phase 2 — Resource metrics. Add a `resourcesOf(pid)` helper (chosen source) and
    fold `{cpu, memMb}` into the `statusOf` result (or a parallel `listServices`
    that includes it, to keep the cheap status path cheap). Expose via CLI first
    (`service status <id>` already prints JSON — add the numbers).
  Phase 3 — Ops HTTP surface. Add the `/api/services/*` routes (placement per Open
    questions). Reuse the kernel's route-table + `sendText` style for consistency.
  Phase 4 — Frontend services page. New route (e.g. `/services`) + a Pinia store
    (`stores/services.js`) mirroring `stores/nodes.js`: list with a status dot
    (reuse the CLI's ●/○ idea), per-row start/stop/restart buttons, a log pane that
    tails `…/logs`, and a small RAM/CPU readout (number + a sparkline). Poll every
    1-2s (transport A).
  Phase 5 — Crash-restart/health. Add `superviseServices()` to the scheduler loop;
    honor `autoRestart` with backoff + `restartCount`; render `crash-looping` red in
    the page. (Optional) per-service `healthCheck` HTTP probe.
  Phase 6 (later, gated) — convert the digitalocean MCP (and/or a new ocraft
    self-MCP) to HTTP/SSE transport so it becomes a truly supervised, health-checked,
    shareable service; add a `healthCheck` URL to its config.

## Open questions / risks

  - Ops routes: separate `runtime/opsServer.js` (honors "kernel depends on nothing
    in runtime") on its own port + a Vite proxy entry, vs. folding into
    `kernel/api.js` (one server, but breaks the layering and makes the kernel import
    the supervisor). RECOMMENDED: separate ops server. CONFIRM.
  - RAM/CPU accuracy: leader-pid-only (simple, undercounts npm/vite trees) vs
    tree-sum. CONFIRM the v1 bar.
  - `pidusage` dependency vs. shelling out to `ps`. Framework-free leaning argues
    `ps`; cross-platform correctness argues `pidusage`. CONFIRM.
  - Auto-restart vs. deliberate stop must never fight; the crash/stop distinction
    rides on the existing `status` field — make sure `stopService` always wins.
  - Stop semantics for MCP standalone services: stdio servers waiting on stdin exit
    cleanly on SIGTERM; verify none ignore it.

## Monetization

No direct angle, and saying so plainly: a single-user local process dashboard isn't
something an individual pays for — `pm2`/`htop`/Activity Monitor already exist free.
The ONLY conditional path is downstream: if ocraft ever ships as a hosted
"scriptable node OS," per-tenant process supervision with resource quotas becomes
table-stakes infrastructure you'd meter (CPU-seconds / RAM-hours per workspace) —
but that's the sandbox+hosting roadmap item earning the money, not this UI. Don't
build it expecting revenue; build it because the headless Direction needs its
processes to stay up.
