REJECTED / OUT-OF-SCOPE — don't re-propose
==========================================

Ideas considered and deliberately set aside. Listed here so feature-generation passes (and
AI dev plans) don't resurface them. When an idea gets cut, add it here with a one-line *why*;
move an item back up to the Roadmap (README) only if its stated reason no longer holds.

- Graph view of the node web — rejected (not wanted). The parentId tree + derived backlinks
  cover the need; a force-directed/visual graph of nodes is not a goal.

- Dependency-aware scheduling / stored `depends-on` (typed) edges — parked (not now). Only
  earns its place once there are real multi-step task chains (e.g. fetch-email -> summarize
  -> post-digest). The cost of a stored edge model isn't justified before then — revisit when
  such chains actually exist. (Backlinks, which need no stored edges, are kept on the Roadmap.)

- World mirror — real-world objects as nodes (digital twins / "GTA5 for real life") —
  rejected (off-identity). Mirroring external things (a Telegram chat, a game, a wearable) as
  nodes is the opposite of ocraft's point: a mirror node is passive data with no behavior — it
  doesn't get crafted, composed (x.x), or run, and it contradicts "data and behavior share one
  tree." It also drags the project onto a connector treadmill (one bespoke, breakable
  integration per source — PSN unofficial, Apple has no API), competing on integrations instead
  of craft. And it's the *less* novel angle — Plan 9 + FUSE already did "mount the world as
  objects" decades ago, whereas ocraft's distinctive lineage is "everything is a live object you
  craft and run" (Lisp/Smalltalk/Emacs). The first slice (a tg-chat node + /api/telegram/* bridge
  + tg-* terminal commands) was built and removed — the endpoints were also an unauthenticated
  user-account read/send hole on localhost. The only salvageable kernel: external data as input
  to a crafted `script` node (pull a source via the API, then *do* something with it) — needs no
  node type and no persistent "twin", so build it only when a concrete use appears, not on spec.
