## What & why
-------------------------------------------------------------------------------
The roadmap wants a Flutter mobile app "controlled / driven from cloud code" —
trigger actions, push state, receive commands. Decode that against what ocraft
actually is: the phone is a second CLIENT of the WS exchange. The "cloud code" that
drives it is the local backend (scheduler + chat-node brain) publishing events and
answering commands through the hub. The phone is BOTH a remote control (send a
`cmd.*` req -> backend runs a task) AND a live dashboard (subscribe to events ->
see services/executions/notifications) AND a quick-capture pad (pub a note ->
backend creates an html node = "remember this").

Why a phone at all: the node OS lives on a laptop that's often closed. The two
things you genuinely want from a pocket are (1) capture a thought into the memory
organ the instant you have it, and (2) get pushed a scheduler/agent notification
("reminder fired", "deploy finished", "the brain wants a decision"). Everything
else (rich editing) is better on the laptop.

## Fit with the node model / architecture
-------------------------------------------------------------------------------
    Flutter app  <--WSS-->  ws-exchange (Deno Deploy)  <--WSS-->  local backend
                                                                    |
                                                            kernel/api.js (truth)

The phone never holds truth and never reaches :3001. It speaks the SAME envelope
the editor speaks (defined in ws-exchange/README.md): `sub` to topics for live
state, `pub`/`req` to ask the backend to do something. A captured note becomes a
real html node only when the backend handles the req and calls POST /api/nodes —
so capture is durable in the exact same store as everything else, addressable at
/node/:id. No separate mobile datastore, no sync engine to reconcile.

This is the FIRST non-editor client, which is the real test of whether the
exchange protocol is actually a contract and not just "what the editor happens to
send." Good forcing function.

## App scope — pick the MVP honestly
-------------------------------------------------------------------------------
Three candidate jobs, in value order:

  1. QUICK-CAPTURE to a node (highest value, smallest surface). A text box + send.
     The note lands as an html node under a `inbox`/`notes` category. This is the
     "remember organ from my pocket" use case and it's genuinely useful day one.
  2. RECEIVE scheduler / agent NOTIFICATIONS (push). The brain or a scheduled task
     wants to tell you something. Two delivery paths — see Connection model.
  3. CONTROL / MONITOR services (lower value on a phone). A list of services with
     status + start/stop/restart buttons that send `cmd.*` reqs. Cool demo, but
     "restart a service from my phone" is a rare real need; park most of it.

MVP = (1) + a thin slice of (2). (3) is a fast-follow once the exchange's `cmd.*`
RPC path is proven. Be honest: monitoring 4 services from a phone is a demo, not a
daily driver.

## Connection model + auth
-------------------------------------------------------------------------------
CONNECTION: Flutter has a first-class `WebSocketChannel` (package:web_socket_channel)
— connect to wss://stream.x8.deno.net/ws, speak the JSON envelope. Unlike the
browser editor, a native app CAN set headers if we later move auth to a header; for
now it uses the same query-param/subprotocol token mechanism the deno-deploy plan
chose (one mechanism for all clients keeps the relay simple).

AUTH: the WS_EXCHANGE_TOKEN gate (deno-deploy plan §3) applies. A native app can
store the token in the platform secure store (flutter_secure_storage / Keychain /
Keystore) — strictly better than the browser editor, where it's a semi-public Vite
env. So the phone is the LEAST leaky client. Still a shared secret, still advisory
role identity — note it; real per-device auth (server-minted tokens, device
revocation) is the future-work item and is exactly the feature a hosted tier would
sell.

THE SLEEPING-LAPTOP PROBLEM (the crux for mobile): if the backend isn't connected
to the hub, a `req` from the phone goes nowhere and live topics are silent. Three
honest options:
  (a) Accept it: the phone only works when the laptop is awake + connected. Fine
      for capture-while-at-desk, useless on the move. Weakest.
  (b) Relay retains last-known state + queues commands (the "last event cache" /
      small inbox flagged in the ws-exchange plan's Open Qs). Phone reads last
      state immediately; queued `cmd`s drain when the backend reconnects. Best
      bang-for-buck; keeps the backend as the only thing that touches truth.
  (c) Stand up an always-on backend (the DO droplet question in the roadmap) so
      there's always a peer on the hub. Heaviest; ties into the "does the droplet
      still earn its place" decision.
  Recommend (b) for capture/notify; revisit (c) only if always-on control matters.

PUSH NOTIFICATIONS: a WS only delivers while the app is foregrounded/connected.
True background push needs FCM/APNs, which needs a sender — i.e. a small cloud
function or the relay calling FCM. MVP: in-app live notifications over the WS while
open (no FCM). Real push = a fast-follow that wires the relay (or a tiny Deno
endpoint next to it) to FCM. Don't promise lock-screen push in the MVP.

## Implementation plan (phased)
-------------------------------------------------------------------------------
PHASE 0 — protocol client (shared)
  - Flutter package: an ExchangeClient (connect, token from secure store, reconnect
    w/ backoff, sub/pub/req helpers over the envelope). Mirror the JS
    useExchange.js so both clients implement one documented contract.
PHASE 1 — quick-capture MVP
  - One screen: text field + Send -> `req cmd.create-note {text, parent?}`.
  - Backend (the exchange service handler) handles cmd.create-note -> POST
    /api/nodes (html node) -> `res` ok with the new /node/:id. Optimistic UI;
    confirm on res.
  - Connection/auth setup screen: paste the WS_EXCHANGE_TOKEN once -> secure store.
PHASE 2 — live notifications (in-app)
  - Subscribe a `notify` topic. The scheduler / a task / the chat-node brain pubs to
    it ("reminder: standup in 10", "deploy finished", "decision needed"). Render as
    a feed + a local OS notification while foregrounded.
PHASE 3 (fast-follow, optional) — service monitor + real push
  - Subscribe "services"; render status; buttons send `cmd.service {id,action}`.
  - Wire FCM/APNs for background push via a small cloud sender. This is the heavy
    bit — scope it separately.

## Open questions / risks
-------------------------------------------------------------------------------
  - Flutter is a WHOLE new toolchain (Dart, Android Studio/Xcode, signing, store
    accounts) for a single-user tool. That's the biggest cost and it's real — weigh
    it hard against the PWA alternative below.
  - The sleeping-laptop problem dominates the design; without relay retention (opt b)
    the app is desk-only. Settle that in the ws-exchange plan first.
  - Real background push = FCM/APNs = more cloud surface + platform setup. MVP must
    not promise it.
  - Shared-token auth means any token holder can act as the phone — fine for a
    personal device, blocks any multi-user/shared future until per-device tokens.

## The lighter alternative — weigh it FIRST (a PWA quick-capture)
-------------------------------------------------------------------------------
For the MVP (quick-capture + in-app notify), a Progressive Web App is dramatically
cheaper than Flutter and reuses the stack ocraft already has (Vite + the editor's
useExchange.js). A PWA:
  + same language/toolchain as frontend/ — zero new build system, zero stores.
  + installable to the home screen, runs the same WS client, captures notes.
  + CAN do background push via the Web Push API + a service worker (no FCM SDK).
  - weaker OS integration (no share-sheet target on iOS until recently, flakier
    background), and iOS Web Push has caveats.
Flutter earns its place ONLY if you want: a real share-sheet capture target ("share
to ocraft" from any app), rock-solid background push, or native widgets. If the goal
is "type a thought into my node OS from my phone + get pinged," start with the PWA,
and treat Flutter as a later upgrade if the PWA's mobile rough edges actually bite.
RECOMMENDATION: ship the PWA MVP first; revisit Flutter as a v2 only if a concrete
native need (share-sheet / reliable push) shows up. This keeps the roadmap item
honest rather than committing to a new toolchain for a capture box.

## Monetization
-------------------------------------------------------------------------------
No direct angle for a personal app — and inventing one would be dishonest. The only
realistic path is the same hosted story as the exchange plan: if ocraft ever became
"a self-hostable personal-agent OS for other people," the phone client + a managed,
authenticated relay is the consumer-facing surface you could charge a small monthly
fee for (because the value to a non-technical user is *not* running their own Deno
relay + handling auth). Cheapest viable charge: bundle "hosted relay + push +
per-device auth" into a low single-digit $/mo tier. Not now; the MVP is for one
user (you), where the cost to charge exceeds any revenue.

## Effort
-------------------------------------------------------------------------------
  PWA route (recommended MVP):  S–M. Reuses Vite + useExchange.js; capture screen +
    service worker for install/push ~ 1–2 days once the exchange protocol exists.
  Flutter route (full):  L. New Dart/Flutter toolchain, ExchangeClient, 2–3 screens,
    secure-store token, platform signing/store setup, then FCM/APNs for real push.
    Realistically a week+ of mostly-toolchain work, most of it not ocraft-specific.
  Either way the TRUE prerequisite is the ws-exchange protocol (Phase 0/1 of that
  plan) — build that first; the phone is thin on top of it.
