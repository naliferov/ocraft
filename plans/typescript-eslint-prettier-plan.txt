===============================================================================
 PLAN — TypeScript + Prettier + ESLint (strict code style) across ocraft
 (written 2026-06-17; for later review)
===============================================================================

GOAL
----
Introduce strict, automated code-style + type safety across the repo:
  - PRETTIER  — one formatter, zero style debates, runs on save / pre-commit.
  - ESLINT    — strict lint that also ENCODES the repo's existing CLAUDE.md rules
                (no single-letter names; no boolean flag params; ESM-only).
  - TYPESCRIPT — type safety, adopted incrementally (no big-bang rewrite).


-------------------------------------------------------------------------------
0. WHAT THE REPO ACTUALLY IS (scope-defining facts — gathered, not assumed)
-------------------------------------------------------------------------------
  - Polyglot, multi-package, ESM everywhere ("type":"module"; CLAUDE.md bans .mjs):
      root/        Node CLI (cli.js, claude-yolo.js) + root deps
      backend/     Node, no framework (scheduler/executor/API server)
      frontend/    Vue 3 + Vite 6 (naive-ui, pinia, vue-router, p5, tone)
      mcp-servers/ digitalocean|gcal|gmail|telegram — each its own Node package
      js-engine/   tiny experiment (yo.js + program.yo)
      ws-exchange/ DENO/TypeScript service (the relay) — SEE §1 EXCLUSION
  - ~45 project .js files total (excluding node_modules/dist/data) — SMALL surface.
    A TS migration here is genuinely tractable, not a mega-project.
  - Exactly ONE .ts today: the Deno relay (backend/transport/ws-exchange-point/
    main.ts; being moved to ws-exchange/ per the Deno plan).
  - NO existing eslint / prettier / tsconfig anywhere — greenfield tooling.
  - Node v22.11.0 installed, no .nvmrc — see §4 (matters for running .ts natively).

  CRITICAL EXCLUSIONS (must be in every tool's ignore list):
  --------------------------------------------------------------------------
  * backend/data/nodes/**/script.js  (6 files today, grows) — these are SANDBOXED
    USER DATA: script-node bodies executed at runtime in the editor, not project
    source. Linting/formatting/type-checking them is wrong — they have their own
    runtime surface (the x.ui / x.* sandbox globals) and are authored as data.
    EXCLUDE from ESLint, Prettier, and tsconfig.
  * backend/data/**/content.html, assets, catalogs — data, not source.
  * backend/executions/**, backend/state/**  — runtime output (already gitignored).
  * frontend/dist/**, **/node_modules/**  — build output / deps.
  * ws-exchange/** (the Deno service) — DIFFERENT RUNTIME. It uses JSR imports
    (jsr:@hono/hono) and Deno globals (Deno.serve, Deno.upgradeWebSocket) that
    Node-oriented ESLint/tsconfig will flag as errors. Lint/format/type-check it
    with DENO'S OWN tooling instead: `deno lint`, `deno fmt`, `deno check` (configure
    in ws-exchange/deno.json). Keep the two toolchains separate; do not let the
    Node ESLint config reach into ws-exchange/.


-------------------------------------------------------------------------------
1. RECOMMENDED OVERALL SHAPE — single root toolchain, per-area overrides
-------------------------------------------------------------------------------
Because every package is ESM and the surface is small, run ONE root-level
toolchain (root devDependencies + root eslint.config.js + root .prettierrc) that
covers root/backend/mcp-servers/js-engine/frontend with per-directory overrides,
rather than duplicating config in each package. Exceptions:
  - frontend keeps its own tsconfig.json (Vue/Vite/DOM settings differ from Node).
  - ws-exchange uses Deno tooling, not this set (see §0 exclusion).

This keeps one source of truth for style while respecting that the frontend
(browser/Vue/DOM) and the backend (Node) have different globals and TS targets.


-------------------------------------------------------------------------------
2. PRETTIER (do this first — formatting is the cheap, high-value win)
-------------------------------------------------------------------------------
  Install (root, dev): prettier
  Files:
    .prettierrc.json   — the style. Note the existing code is NOT uniform:
                         do-api.js uses no semicolons + single quotes; backend
                         entries vary. PICK A STYLE and let Prettier enforce it.
                         CONFIRM: semicolons yes/no, quotes single/double,
                         printWidth (100 suggested), trailingComma "all".
                         (Recommend matching the most-common existing style to
                         minimize churn: single quotes, no semicolons, width 100
                         — that matches do-api.js / the DO MCP.)
    .prettierignore    — the §0 exclusions: backend/data/, backend/executions/,
                         backend/state/, **/dist/, **/node_modules/, ws-exchange/,
                         *.yo, package-lock.json.
  + eslint-config-prettier (turns OFF ESLint rules that conflict with Prettier,
    so the two never fight — formatting is Prettier's job, code-quality is ESLint's).
  Scripts (root package.json):
    "format":       "prettier --write ."
    "format:check": "prettier --check ."
  Frontend: Prettier handles .vue too (built-in). One root Prettier run covers it.

  EXPECT a large one-time diff on the first `prettier --write`. Land it as its own
  isolated commit ("chore: prettier baseline") so real changes aren't buried.


-------------------------------------------------------------------------------
3. ESLINT (flat config; also enforces the CLAUDE.md coding rules)
-------------------------------------------------------------------------------
  Use ESLint flat config (eslint.config.js at root) — the current standard.

  Install (root, dev):
    eslint, @eslint/js, globals,
    eslint-config-prettier,
    eslint-plugin-vue + vue-eslint-parser           (frontend .vue)
    typescript-eslint                                (once TS lands, §4)

  eslint.config.js structure (array of config objects, most-specific last):
    1) Global ignores: the §0 exclusion list (data/nodes, dist, executions,
       state, node_modules, ws-exchange, *.yo, package-lock.json).
    2) Base JS (all .js): @eslint/js recommended + the HOUSE RULES below,
       languageOptions sourceType:"module" (ESM), ecmaVersion latest.
    3) Backend/root/mcp override: languageOptions.globals = globals.node.
    4) Frontend override (frontend/**): eslint-plugin-vue flat "recommended"
       config, parser vue-eslint-parser, globals.browser. Covers .vue + .js there.
    5) (after TS) typescript-eslint recommended/strict for .ts/.vue<script ts>.

  HOUSE RULES — encode the repo's CLAUDE.md conventions as lint, so they're
  enforced, not just documented:
    * NO SINGLE-LETTER NAMES (CLAUDE.md coding rule 1):
        "id-length": ["error", { "min": 2,
            "exceptions": [],                 // start strict; widen only if needed
            "properties": "never" }]          // don't flag object keys you don't own
      This is the rule that makes `const d = …` / `(i) =>` fail — exactly the
      repo's stated convention. Pair with "camelcase" and optionally a
      descriptive-name nudge.
    * NO BOOLEAN FLAG PARAMETERS (CLAUDE.md coding rule 2):
      ESLint has no perfect built-in for "no boolean positional arg". Best effort:
        - "no-restricted-syntax" targeting boolean literals in call arguments
          (heuristic; noisy — may need tuning or scoping to warn).
        - OR accept this rule stays human-enforced / PR-review-enforced and just
          document it. Recommend: WARN via a tuned no-restricted-syntax, don't
          block CI on it. Be honest that it's partial.
    * ESM hygiene: "no-restricted-syntax"/import rules to keep imports clean;
      rely on sourceType:module. (CLAUDE.md "never .mjs" is moot — we never add
      .mjs; new typed files are .ts, plain ones stay .js.)
    * Turn on @eslint/js "recommended" + selected strictness (no-unused-vars as
      error with argsIgnorePattern "^_", no-undef, eqeqeq, prefer-const, etc.).
    * eslint-config-prettier LAST so Prettier owns formatting.

  Scripts (root):
    "lint":     "eslint ."
    "lint:fix": "eslint . --fix"

  FIRST RUN will surface real violations (single-letter vars exist in places).
  Triage: auto-fix what's safe, hand-fix the id-length hits, commit separately.


-------------------------------------------------------------------------------
4. TYPESCRIPT (incremental — no big-bang; pick a track per area)
-------------------------------------------------------------------------------
NODE VERSION NOTE (decides how backend .ts runs):
  Node 22.11 is installed. Native TS "type stripping" (run .ts directly, no build)
  is unflagged only on Node 22.18+ / 24. On 22.11 it needs --experimental-strip-types,
  and enums/namespaces/param-props need --experimental-transform-types.
  RECOMMENDATION: add an .nvmrc pinning Node 24 LTS (or >=22.18) so `node main.ts`
  Just Works for backend/CLI/MCP without a bundler. CONFIRM the Node bump.

Two adoption tracks — different areas warrant different speed:

  TRACK A — "type-check JS in place" (lowest friction; keeps .js, no renames):
    - Add tsconfig.json with: allowJs:true, checkJs:true, noEmit:true,
      strict:true, module/moduleResolution "nodenext", target recent ES.
    - Annotate with JSDoc where useful; TS checks the existing .js for type errors.
    - Script: "typecheck": "tsc --noEmit".
    - Good for: backend, mcp-servers, root CLI — surfaces bugs without churn.
    - This is the recommended STARTING point everywhere: turn the lights on first,
      see what breaks, THEN decide what to rename.

  TRACK B — "migrate to .ts" (full types; do it area-by-area after Track A is green):
    - Rename .js -> .ts, add real annotations, drop checkJs for that area.
    - Run via native Node type-stripping (needs the Node bump above) OR a tiny
      build (tsc/esbuild) if you'd rather not bump Node.
    - Sequence by value & isolation:
        1. mcp-servers/* (small, self-contained, clear I/O types) — easy wins.
        2. backend/lib + api + executor/scheduler/storage (core, benefits most).
        3. backend/entries (many small files; do as touched).
        4. root cli.js / claude-yolo.js.

  FRONTEND (Vue 3 + Vite) — Vite supports TS out of the box; the path is well-trodden:
    - Add frontend/tsconfig.json (extends a Vue base; DOM + bundler resolution,
      strict:true) + frontend/tsconfig.node.json for vite.config.
    - Install: typescript, vue-tsc, @vue/tsconfig, @types/node.
    - Type-check .vue with vue-tsc (tsc can't see SFCs):
        frontend "typecheck": "vue-tsc --noEmit".
    - Migrate SFCs incrementally: <script setup lang="ts"> per component as you
      touch them; .js helpers -> .ts. Vite compiles TS automatically (no build
      change). eslint-plugin-vue + typescript-eslint cover the typed SFCs.
    - Pinia/vue-router are fully typed — good payoff in the store/router layer.

  EXCLUDE from all tsconfigs: the §0 list (esp. backend/data/nodes/**/script.js —
  sandbox data; and ws-exchange/** — Deno, type-checked by `deno check` separately).


-------------------------------------------------------------------------------
5. AUTOMATION — pre-commit + (optional) CI
-------------------------------------------------------------------------------
  - PRE-COMMIT (recommended): husky + lint-staged at root. On commit, run
    prettier --write + eslint --fix on STAGED files only (fast). Keeps the tree
    clean without a full-repo run each time.
      lint-staged: { "*.{js,ts,vue}": ["eslint --fix", "prettier --write"],
                     "*.{json,md,css,html}": ["prettier --write"] }
    NOTE: CLAUDE.md says "develop on main" — a pre-commit hook is the right guard
    since there's no PR-branch gate by convention.
  - CI (optional, if/when GitHub Actions is wanted): a workflow running
    format:check + lint + typecheck (+ frontend vue-tsc) on push. .github/ exists
    already — can add a lint workflow there.
  - Also add VS Code settings (.vscode/settings.json): formatOnSave + ESLint
    fixAll, so the editor enforces it live. (Optional, nice-to-have.)


-------------------------------------------------------------------------------
6. SEQUENCING (small, reversible, each its own commit)
-------------------------------------------------------------------------------
  1. PRETTIER baseline: add config + ignore; run --write; commit alone
     ("chore: prettier baseline"). Biggest diff, lands once.
  2. ESLINT: flat config + house rules (id-length etc.) + scripts; --fix the
     safe stuff; hand-fix id-length violations; commit.
  3. eslint-config-prettier wired so the two don't fight (part of step 2).
  4. TS Track A (checkJs) on backend + mcp + root: add tsconfig, "typecheck"
     script; fix what it surfaces; commit. (Frontend gets its own tsconfig +
     vue-tsc here.)
  5. TS Track B migration, area by area (mcp-servers -> backend core -> entries
     -> cli -> frontend SFCs as touched). Each area = its own PR-able commit.
  6. .nvmrc (Node bump) if going native-.ts for backend (step 5 prerequisite).
  7. husky + lint-staged pre-commit; optional CI workflow.
  Steps 1-2 deliver most of the value fast; 4-5 are the longer tail.


-------------------------------------------------------------------------------
7. DECISIONS TO CONFIRM
-------------------------------------------------------------------------------
  [1] Prettier style: semicolons? quotes? printWidth? trailingComma?
      (Rec: no semicolons, single quotes, width 100, trailingComma "all" —
       matches the existing DO MCP / do-api.js style; least churn.)
  [2] "No boolean flag params" rule: enforce as ESLint WARN (tuned heuristic) or
      leave human-enforced? (Rec: warn-only; it's not cleanly auto-detectable.)
  [3] TS end state: type-check-JS-only (Track A, keep .js) OR full migrate to .ts
      (Track B)? (Rec: A everywhere now, B area-by-area over time.)
  [4] Node bump to 24 LTS / >=22.18 (enables native `node main.ts`)? Needed only
      if backend goes to real .ts without a build step.
  [5] Pre-commit hook (husky+lint-staged) yes/no; CI lint workflow yes/no.

  REMINDER: ws-exchange/ (Deno) is handled by `deno lint`/`deno fmt`/`deno check`,
  configured in its deno.json — NOT by this Node toolchain. And
  backend/data/nodes/**/script.js stays excluded everywhere (sandbox user data).
