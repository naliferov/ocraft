## What & why
ocraft has no behavior gate. The autonomous dev loop (plans/autonomous-dev-loop-
plan.txt) can edit a node and open a PR, but the only checks today are
`node --check`, `npm run build`, and a claude-in-chrome screenshot — none of
which prove a node still *does what it should*. That is the load-bearing caveat
in the dev-loop plan ("NO REAL TEST GATE"): every PR genuinely needs a human
until an acceptance gate exists. This harness is that gate.

The key realisation: a node script is ALREADY a runnable program in the browser.
`runNodeCode(code, selfId, ui)` (frontend/src/composables/useNodeScripts.js:226)
blob-imports `export default (x) => {...}` and hands it an `x` context whose
`x.x("id")` runs any other node by id. So an acceptance check is not new
machinery — it's just *another node* that runs the node-under-test and asserts on
the result. Make the check a node and it inherits everything: authored in the
editor, versioned, composable via `x.x`, and runnable headlessly later by the
same `x` context (minus `x.ui`).

Non-goal: a general JS unit-test framework for the repo's own .js files (that's
Vitest/node:test, a separate concern). This harness tests *nodes* — the user's
content/programs in the store — which is the thing the dev loop edits.

## Fit with the node model / architecture
Pure frontend, reusing existing runtime surfaces:
  - Execution: `runNodeCode` / `x.x` already exist (useNodeScripts.js). The check
    body calls `await x.x("<target-id>", [args])` exactly like any composing
    script does today.
  - The `x` context is built by `makeCtx` (useNodeScripts.js:189). The harness
    adds ONE field — an assertion surface — to the ctx handed to a check node.
    Everything else (`x.x`, `x.auth`, `x.vue`, `x.log`) is unchanged.
  - UI: `createScriptUi` (useScriptUi.js) already renders a log panel above the
    editor with up/down/sys colored lines — perfect for a green/red pass list.
    The harness panel reuses it rather than inventing a results widget.
  - Storage: a check is a node body fetched/saved via the same
    GET/POST /api/nodes/:id/body path. No new kernel/api.js endpoint required for
    the MVP (see Phase C for the optional headless gate, which does touch backend).

This keeps the harness honestly inside the node model: "a note that is a program"
extended to "a note that *verifies* a program."

## Where checks live — decision
Two options; pick (A) for the MVP.

(A) A `test` node type. A check is its own node, `type: "test"`, whose body is
    `export default async (x) => { ... }`. It declares its target(s) by calling
    `x.x("<id>")`. Pros: zero schema change (the store is type-agnostic —
    `POST /api/nodes` takes any `type`), checks are first-class nodes you can
    list/search/run, and a node can have many test children under it. Cons: the
    target↔test link is implicit (the id inside the body) unless we add a
    convention (e.g. the test node's `parentId` is the node it tests, or a
    `state.json` `targetId`).

(B) A paired `tests` body on the script node itself (a second body, like the
    "Multiple bodies/attachments per node" roadmap item). Cleaner colocation, but
    multi-body nodes don't exist yet — this would front-load that feature. Defer.

Decision: (A) `test` node type for the MVP. Convention: a test node lives as a
CHILD of the node it tests (so `node.parentId` = target id), AND/OR the body
names the target via `x.x("id")` — the harness discovers "tests for node N" by
scanning child nodes of type `test`. Revisit (B) once multi-body nodes land.

## The assertion surface — decision
Add `x.expect` (and a thrown-on-fail `x.assert` for terse checks) to the check
node's ctx. Minimal, dependency-free, BDD-ish so it reads well in a body:

    export default async (x) => {
      const sum = await x.x("5", [2, 3])      // run node 5 with args
      x.expect(sum).toBe(5)
      x.expect(await x.x("5", [0, 0])).toBe(0)
      x.assert(sum > 0, "sum must be positive")   // terse alternative
    }

`x.expect(value)` returns a matcher object: `toBe`, `toEqual` (deep), `toThrow`
(value is a fn/thunk), `toContain`, `toBeCloseTo`. Each matcher, on failure,
throws an Error with a readable message; the harness catches it, marks the check
failed, and records message + the expected/actual. A check with no throw =
passed. `x.assert(cond, msg)` is sugar for `if (!cond) throw new Error(msg)`.

Keep it a single ~40-line module — NOT a vendored matcher lib — so the surface
stays small and mirror-able backend-side (same reasoning as the `x` shape staying
simple for a future headless runner).

## Implementation plan (phased) — concrete steps, contracts, real files

PHASE A — the assertion surface + run-one-check (no UI yet)
  1. New module: frontend/src/composables/useNodeExpect.js — exports
     `createExpect()` -> `{ expect, assert, results }` where `results` is an array
     the matchers push pass/fail records into:
        { ok: boolean, matcher, expected, actual, message }
     `expect(actual)` returns matchers that compare and push a record (throwing on
     fail so the check body stops at the first failure, jest-style). `assert`
     pushes + throws likewise.
  2. Wire into ctx: in `makeCtx` (useNodeScripts.js), accept an optional
     `expectApi` param and spread `...(expectApi ? { expect: expectApi.expect,
     assert: expectApi.assert } : {})`. Top-level check runs pass it; nested
     `x.x` calls do NOT (a check asserts; the nodes it calls just run) — same
     pattern as `ui` being top-level-only.
  3. New runner: `runNodeCheck(checkNodeId)` in a new
     frontend/src/composables/useNodeTests.js — fetches the check body via
     loadModule (or runNodeCode for unsaved), builds a ctx with a fresh
     createExpect(), runs `mod.default(ctx)`, and returns
        { id, name, passed: results.every(r=>r.ok) && !threw, results, error }
     Catch the thrown matcher/runtime error so one failing check never breaks the
     batch.
  4. Verify (console first): pick a deterministic node — e.g. node 5 (the x.x
     callee in the README example) — author a `test` child whose body asserts on
     `x.x("5")`, call runNodeCheck from a scratch script, see pass→green / break
     the assert → red record. WASM nodes (10/4): a check can `x.x("10")` and
     assert on `add(2,3)===5`.

PHASE B — the harness panel (run all checks for a node / the store)
  5. Discovery: in useNodeTests.js add `findChecks(targetId)` = store.nodes filter
     type==='test' && parentId===String(targetId); and `allChecks()` = all
     type==='test'. (Mirror the callableOptions computed pattern in Script.vue.)
  6. `runChecks(ids)` — map ids through runNodeCheck, concurrently (Promise.all)
     or sequentially (sequential is safer if checks mutate the store; default
     sequential, note it). Returns the array of result objects.
  7. UI: a harness panel. Reuse createScriptUi's log surface for the result list
     — push one line per check: `direction:'down'` (green) for pass, a red line
     for fail with the matcher message. Add a "Run tests" button next to Run in
     Script.vue's controls (Script.vue:140) that appears when the open node HAS
     child checks, plus a top-level "Run all tests" entry (a command-palette
     action later, a button for now). Summary line: `N passed, M failed`.
  8. A dedicated `test` node renderer (frontend/src/components/NodeItem/nodeTypes/
     Test.vue) — register it in NodeItem.vue's type→component map (NodeItem.vue:
     30-36, the same map the keystone wants to make data-driven). For the MVP a
     Test.vue can be a thin wrapper around the Script editor (a check IS a script)
     plus a "Run this check" button and an inline pass/fail readout. Keep it
     minimal; do NOT block on the keystone's dynamic-handler work.

PHASE C — headless gate for the dev loop (the payoff; touches backend)
  9. The dev-task entry (plans/autonomous-dev-loop-plan.txt) needs to run a node's
     checks WITHOUT a browser to gate a PR. Options, cheapest first:
       (i)  A backend check-runner: a `bin/cli.js test <nodeId>` (or a `node-test`
            entry) that re-implements the tiny `x` context server-side enough to
            run a JS check body in a VM/`node:vm` or a worker, with `x.x`
            resolving other node bodies from kernel/data. This is exactly the
            "reimplement the `x` shape backend-side" path the runtime comments
            already anticipate (useNodeScripts.js:59-62). WASM checks work too
            (the backend already compiles AS -> wasm at /api/nodes/:id/wasm).
       (ii) Headless-browser the existing frontend runner via the claude-in-chrome
            MCP / a Playwright page that loads the editor and calls runChecks —
            reuses ALL the frontend code, zero reimplementation, but needs a
            running frontend+api. Good as a stopgap before (i).
     Decision: ship (ii) as the immediate dev-loop gate (no new backend code),
     build (i) when the headless `x` reimplementation is worth it. Exit code /
     JSON result (`{passed, failures[]}`) is what the dev-task gate reads.
 10. Wire the gate: dev-task runs the target node's checks, blocks the PR on all
     green, and pastes the pass/fail summary into the PR body as acceptance
     evidence (replacing today's "screenshot only" evidence for nodes that have
     checks). Nodes WITHOUT checks still fall back to the weak gate — and the loop
     should prefer authoring a check as part of any node change (a norm, not
     enforced).

## Open questions / risks
  - Target↔test linkage: parentId-as-target is clean but means a test can't live
    elsewhere in the tree. Alternative: a `state.json` `targetId` field (the store
    already carries per-node state.json). Decide in Phase B; parentId for MVP.
  - Determinism: checks that call `x.fetch`/network or read wall-clock are flaky.
    Document that acceptance checks should be deterministic; the dev-loop gate
    should treat a flaky check as a failure, not retry-until-green.
  - Module mocking (flagged under "Multi-project & tooling"): to test a node that
    composes others in isolation, the harness needs to stub a dependency's `x.x`
    return. MVP has none — checks run the real dependency graph. A later
    `x.mock("id", fakeFn)` on the check ctx swaps loadModule's result for that id
    within the run. Out of scope for Phase A/B; note the seam in makeCtx.
  - Mutation safety: a check that creates/deletes nodes pollutes the store. Long
    term, run checks against a scratch/transaction store or snapshot+restore
    (ties into "Persistent trash + restore" / "scheduled git-snapshot backup").
    MVP: document "checks should be read-only or self-cleaning."
  - Don't couple to the keystone: Test.vue can be a hardcoded handler today; it
    doesn't need the data-driven-handler work to land first.

## Monetization
No direct angle — a test gate is infrastructure, not a feature anyone pays for
standalone. Its real value is *indirect and large*: it's the unlock that makes the
autonomous dev loop trustworthy enough to run unattended, which IS a sellable
story ("a node OS that safely extends itself"). If ocraft ever becomes a hosted
"scriptable node OS," per-node acceptance checks are also the basis for a
marketplace trust signal (shared handlers/nodes that ship with green checks).
Don't charge for the harness; charge for what it makes safe.

## Marketing
Demoable as the proof behind the autonomous-dev-loop narrative: a clip of an AI
editing a node, the harness panel flashing red, the AI fixing it, then green, then
a PR opening — "my notes app tests itself before it ships a change." Pairs with
the dev-loop content and the YouTube plan. Weaker standalone artifact (a pass/fail
list isn't flashy); its punch comes from being the gate in the loop story.

## Effort — S/M/L
  M overall.
  - Phase A (expect surface + runNodeCheck + makeCtx hook): S. Small, isolated,
    reuses runNodeCode wholesale.
  - Phase B (discovery + harness panel + Test.vue + buttons): M. Most of the work
    is UI wiring and the type registration; the run logic is thin.
  - Phase C (headless gate): S for the Playwright/claude-in-chrome stopgap (ii),
    M–L for a real backend `x` reimplementation (i). Do (ii) first; (i) only when
    the headless runner earns its keep.
