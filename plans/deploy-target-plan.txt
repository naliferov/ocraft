## What & why

Deploy is wired through GitHub Actions. On push to `main`, `.github/workflows/ci.yml`
SSHes into `secrets.DEPLOY_HOST` (a DigitalOcean droplet), writes a one-line `.env`,
`git pull`, `npm install`, then `node bin/cli.js run deploy` (the `deploy` task = a
git pull under the scheduler lock). **It is currently broken**: the droplet behind
`DEPLOY_HOST` was destroyed, so every push fails at the SSH step. `runtime/state/
droplet.json` holds only a `{region, size}` spec — nothing live to SSH into.

`runtime/tasks/do-droplet-up.js` (or the DO MCP `do_create_droplet`) re-provisions a
box end to end: create → wait active/IP → wait sshd → cloud-init → install Node via
NodeSource + git → clone into `/root/ocraft`. But re-running it alone will NOT
restore deploy. Three concrete gaps, then the bigger strategic question.

The README frames the strategic question first and it's the right order: **does an
always-on DO droplet still earn its place** now that the WS exchange moved to Deno
Deploy? So this plan answers that FIRST (a recommendation), then specifies the fix
for whichever path is chosen.

## The strategic question (answer first)

What did the box actually do? Historically two jobs: (1) host the cloud WS exchange,
and (2) be an always-on home for the scheduler so scheduled tasks run when the laptop
is closed. Job (1) is **gone** — the exchange now lives on Deno Deploy
(`wss://stream.x8.deno.net/ws`), serverless, free-tier, zero ops. So the droplet's
remaining justification is **job (2) alone: a persistent host for the scheduler +
its tasks** (telegram-reminder/poll, every-hour, future life-management jobs that
must fire on a cadence regardless of laptop state).

Three options:

  (1) DROP THE PERSISTENT BOX. Run the scheduler only on the laptop (it's already a
      managed service). Cheapest ($0), least ops. Cost: tasks don't fire when the
      laptop is asleep/closed — fatal for a "personal life-management system" whose
      whole point is *the scheduler decides when/what* even when you're not at the
      machine. A reminder that only fires when you're already at your computer is
      pointless. So: acceptable ONLY if you accept the scheduler is best-effort and
      laptop-bound. For the stated Direction, it's not.

  (2) DENO DEPLOY FOR EVERYTHING. Tempting (the exchange already lives there), but
      Deno Deploy is a request-driven serverless edge runtime, not a host for a
      long-running Node daemon with a filesystem. The scheduler is a forever-loop
      that reads/writes node folders and `runtime/state/` on disk and shells out
      (ssh, git, sharp). That is the opposite of an ephemeral edge function. You'd
      have to rewrite the scheduler as cron-triggered stateless invocations against
      external storage — a large rewrite of the runtime, and tasks like
      `img-optimize` (sharp, fs) don't fit at all. NOT a fit
      for the scheduler. Keep using Deno Deploy for what it's good at (the stateless
      WS relay), not as a Node host.

  (3) KEEP A SMALL ALWAYS-ON BOX (RECOMMENDED). The cheapest DO droplet
      (`s-1vcpu-512mb-10gb`, ~$4/mo) is enough to run the scheduler 24/7. It's also
      the natural home for the roadmap's *sandboxed isolated workers* and a future
      always-on *ocraft self-MCP* / chat-node "brain." The box stops being "the WS
      relay host" (Deno Deploy took that) and becomes "the scheduler/automation
      host" — a clearer, smaller job. The `do-droplet-up`/`-down` pair already makes
      it disposable, so you can even run it only when you need persistence and tear
      it down to stop billing.

  RECOMMENDATION: (3) — keep a small always-on droplet, re-scoped to "scheduler +
  automation host," and fix the three deploy gaps below. The Direction *requires*
  an always-on scheduler, $4/mo is noise, and the disposable up/down tasks already
  exist. CONFIRM the recommendation (and whether you want it always-on, or
  on-demand via the up/down tasks).

  (Honest middle path if you want $0 most of the time: keep the deploy machinery,
  but normally leave the box DOWN and bring it up via `do-droplet-up` only for
  stretches when you need the scheduler running unattended. The reserved-IP fix
  below makes that cheap because the address survives the down/up cycle.)

## Fit with the node model / architecture

- The box is just another HOST for `runtime/` — same code, same scheduler service.
  No node-model change; the node store ships with the repo (`kernel/data/nodes/` is
  in git), so `git pull` already syncs the nodes onto the box. (Note: that makes the
  box's node tree a deploy-time snapshot, not a live mirror of the laptop's edits —
  acceptable for scheduled tasks, but be aware the box won't see laptop-side node
  edits until the next push. CONFIRM this is fine, or the box needs to be the node
  source of truth, which is a bigger design call.)
- Reserved IP, droplet lifecycle, and env provisioning all belong in `runtime/tasks/`
  + the DO MCP — consistent with `do-droplet-up`/`-down` already living there and
  sharing `do-api.js`.

## The three gaps (README), specified

  GAP 1 — STABLE ADDRESS (reserved IP). A fresh droplet gets a new public IP, so
    `DEPLOY_HOST` changes every rebuild and the SSH step breaks again. Fix: a DO
    **reserved IP**, assigned to the box on create, so the address survives rebuilds.
    The DO MCP has the tools (`do_create_reserved_ip`, `do_assign_reserved_ip`,
    `do_list_reserved_ips`, `do_unassign_reserved_ip`). Implementation:
      - One-time: create a reserved IP in the droplet's region; record it (a new
        `runtime/state/reservedIp.json`, or extend `droplet.json`).
      - In `do-droplet-up`: after `waitForActive`, `do_assign_reserved_ip` the
        reserved IP to the new droplet id; use the RESERVED IP (not the ephemeral
        one) for the subsequent ssh/clone steps and as the saved `ip`.
      - In `do-droplet-down`: optionally `do_unassign` before delete (delete frees
        it anyway), but DO NOT release the reserved IP — keeping it is the point.
      - Set `secrets.DEPLOY_HOST` (GitHub) to the reserved IP, permanently.
    CAVEAT: a reserved IP that's not attached to a running droplet bills a small
    hourly fee (DO charges for *idle* reserved IPs). Cheap, but real — note it if you
    run the box on-demand. CONFIRM the region is pinned (the reserved IP is
    region-locked; `do-droplet-up` defaults `fra1`).

  GAP 2 — NODE-PATH MISMATCH. `ci.yml` hardcodes an nvm path
    (`export PATH=/root/.nvm/versions/node/v25.8.2/bin:$PATH`), but `do-droplet-up`
    installs Node via NodeSource/apt, landing at `/usr/bin/node` (on `$PATH`
    already). So the CI's PATH export points at a directory that doesn't exist on a
    box provisioned by `do-droplet-up` → `node`/`npm` not found → deploy fails.
    Align ONE side — pick the NodeSource/apt reality since that's what provisions the
    box:
      - Edit `ci.yml`: DROP the nvm PATH line entirely (NodeSource puts node on the
        default PATH). The SSH command becomes `cd /root/ocraft && … && git pull &&
        npm install && node bin/cli.js run deploy` with no PATH munging.
      - Pin the Node MAJOR for reproducibility: `do-droplet-up` uses
        `setup_current.x` (the bleeding-edge "current" line) — pin to a specific LTS
        (e.g. `setup_22.x` / `setup_24.x`) so the box's Node doesn't silently jump a
        major between rebuilds and break native deps (sharp). CONFIRM the target
        major (match the laptop dev Node).
    DELETE the stale `v25.8.2` reference so nothing re-introduces the nvm assumption.

  GAP 3 — NO SCHEDULER PROVISIONING. `do-droplet-up` clones the repo but never
    (a) writes a persistent `.env`, nor (b) starts the `scheduler` service — so even
    a successfully deployed box runs no scheduled tasks. Fix, after the clone step in
    `do-droplet-up` (and mirrored in `deploy` for redeploys):
      - WRITE A PERSISTENT `.env` on the box from the needed secrets (currently
        `TELEGRAM_BOT_TOKEN`; whatever tasks require). `ci.yml` already writes a
        one-line `.env` on each deploy, which CLOBBERS anything else — make it
        additive or template a full `.env` so provisioning and deploy agree on its
        contents. CONFIRM the secret set the box needs (telegram at least;
        DO/google creds only if those tasks run on the box).
      - `npm install` on the box (already in ci; ensure do-droplet-up does it too so
        a freshly provisioned box is runnable before the first push).
      - START THE SCHEDULER as a service: `node bin/cli.js service start scheduler`.
        For TRUE persistence across reboots/crashes, the service manager's detached
        spawn isn't enough (it dies with the box on reboot) — install a **systemd
        unit** that runs the scheduler on boot (and, per the process-management plan,
        let systemd own *just* the scheduler; the scheduler then supervises/auto-
        restarts the rest). A `runtime/` helper task (`provision-scheduler`) can
        drop the unit file and `systemctl enable --now` it. CONFIRM: systemd unit
        (durable, survives reboot — RECOMMENDED) vs. plain `service start` (simpler,
        dies on reboot).

## Implementation plan (phased)

  Phase 0 — DECIDE. Confirm the recommendation (keep a small box) vs drop it. If
    dropped, stop here: delete `ci.yml`'s deploy step (or gate it off) and accept a
    laptop-bound scheduler; the rest of this plan is moot.
  Phase 1 — Reserved IP (GAP 1). Create the reserved IP once (DO MCP); store it;
    teach `do-droplet-up` to assign+use it; set `DEPLOY_HOST` to it. Verify
    `ssh root@<reservedIp>` after an up.
  Phase 2 — Node path (GAP 2). Strip the nvm PATH line from `ci.yml`; pin the Node
    major in `do-droplet-up` (`setup_<major>.x`). Verify `node -v`/`npm` resolve
    over SSH on a fresh box.
  Phase 3 — Provisioning (GAP 3). Add `.env` templating + `npm install` + scheduler
    start to `do-droplet-up`; add a systemd unit (a small `provision-scheduler`
    helper) for reboot persistence. Verify the scheduler is up and a job fires.
  Phase 4 — Green deploy. Push to `main`; confirm `ci.yml` reaches the box, pulls,
    installs, runs `deploy`, and the scheduler keeps running. (Optional: have the
    deploy task `service restart scheduler` post-pull so new code is picked up.)
  Phase 5 (optional) — On-demand mode. Document the down/up cycle (reserved IP makes
    it cheap); optionally a tiny task to bring the box up for N hours then down.

## Open questions / risks

  - Reserved IP region-lock: pin `do-droplet-up`'s region to the reserved IP's
    region (defaults `fra1`); a mismatch makes the assign fail. CONFIRM region.
  - Idle reserved-IP billing: small hourly charge when unattached — note it for the
    on-demand path.
  - `.env` clobber: ci.yml's single-line `.env` overwrites; reconcile with the
    provisioned `.env` so secrets aren't lost on each deploy.
  - Node major drift: `setup_current.x` can jump majors and break sharp's prebuilt
    binaries; pin a major.
  - Node tree freshness: the box's nodes are a git snapshot, not a live mirror of
    laptop edits — fine for scheduled tasks, surprising if you expect the box to see
    your latest note. CONFIRM acceptable.
  - SSH key: `do-droplet-up` injects key id 53064860 ("macbook air"); `ci.yml` uses
    `secrets.DEPLOY_SSH_KEY`. Ensure the CI key's public half is on the box (either
    the same key, or add the CI key to the droplet on provision).
  - Reboot persistence needs systemd; the bare `service start` dies on reboot.

## Monetization

No direct angle, stated plainly: this is internal infra (restore my own deploy), not
a feature anyone pays for. The only honest adjacency is that getting a reliable,
disposable, reserved-IP'd always-on host working is the PREREQUISITE for two future
items that *could* monetize — sandboxed resource-capped worker execution (meterable
CPU/RAM-hours, the real revenue candidate) and a hosted multi-tenant ocraft. So fund
this as the foundation those need, not as a money-maker itself. Pricing realism: a
single $4/mo droplet is a personal cost, not a product.
