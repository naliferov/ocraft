DAILY POSTGRES BACKUP — prod (ocraft.dev)
=========================================

Goal: a daily, restorable, off-box backup of the prod `ocraft` database.

Built with ocraft's OWN runtime instruments — a TASK on the SCHEDULER, not a system cron +
shell script. Dogfoods the runtime, reuses the executor's scheduling / execution-record /
ctx.log / timeout machinery, and keeps one automation model. See AGENTS.md ("automate through
ocraft's own runtime instruments").

STATUS: live on prod (committed + deployed) and restore-verified (200 nodes / 173 bodies).


PIECES
------
- runtime/tasks/pg-backup.js — the task: pg_dump -Fc (via DATABASE_URL over TCP) -> verify the
  archive (pg_restore --list) -> offsite (if a remote is set) -> rotate -> log via ctx.log.
  Throws on failure, so the execution record is marked `error`.
- runtime/scheduler.js jobs[] — { id:'pg-backup', task:'pg-backup', schedule:{hour:3,minute:30} }
  (daily 03:30 server-local; the prod box is UTC).
- the `scheduler` SERVICE must be RUNNING:  node runtime/cli.js service start scheduler  (it is).

DUMPS: <home>/ocraft-backups/<db>-<UTCstamp>.dump (custom format; prod root = /root/ocraft-backups).
RETENTION: drop dumps older than 30 days, by mtime. The just-written dump is always kept and
rotation only runs after a fresh dump exists, so it can never reach zero backups. Overrides:
OCRAFT_BACKUP_KEEP_DAYS (default 30), OCRAFT_BACKUP_DIR.


RESTORE  (on the box as root — reads /root, connects over TCP via DATABASE_URL)
-------
  set -a; . /root/ocraft/.env; set +a
  pg_restore --clean --if-exists --no-owner -d "$DATABASE_URL" <file.dump>


IMPROVEMENTS (planned, not built) — ordered by value
----------------------------------------------------
1. OFFSITE copy — the big one: a local-only dump dies with the droplet. Enable the (already-wired)
   offsite step by setting OCRAFT_BACKUP_RCLONE_REMOTE (an rclone Drive or S3/Spaces remote) in the
   scheduler's env; on prod add it to the CI-written .env so it survives deploys.
     (a) Google Drive via rclone — free, already used here (one-time `rclone authorize`).  <- recommended
     (b) DigitalOcean Spaces — native, S3-compatible, ~$5/mo.
     (c) Pull-to-Mac — a cron on the laptop rsyncs dumps down; box stays credential-free.
2. SUPERVISE the scheduler — backups now depend on the `scheduler` service staying up, and services
   aren't auto-restarted on crash/reboot yet (README roadmap: `autoRestart` declared, not acted on).
   Give it a systemd unit (or act on autoRestart) so a reboot doesn't silently stop backups.
3. STALENESS ALERT — warn (Telegram via the bot already in .env) if the newest dump is > 25h old —
   i.e. a backup silently didn't run. The "did it actually run?" alarm, separate from per-run errors.
4. AUTOMATE the monthly restore-check — today it's manual (restore newest into a throwaway db, count
   nodes/bodies). Make it a second task/job so "it actually restores" is proven without a human.


VERIFY CADENCE
--------------
- Every run: verifies the archive is well-formed (pg_restore --list).
- Monthly (manual for now — see improvement 4): restore newest into a throwaway db, count rows.
