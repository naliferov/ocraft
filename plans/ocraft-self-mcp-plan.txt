## What & why
-------------------------------------------------------------------------------
Every other MCP gives Claude hands on an EXTERNAL service (Telegram, Gmail, GCal,
DO). ocraft has none on ITSELF. So an AI can edit ocraft's source files (via an
interactive Claude Code session's Read/Grep/Edit, or the ai-chat node) but cannot
OPERATE the running system: it can't list scheduled jobs, run a task, check a
service, tail an execution log, or — most importantly — search and append to the
notes-as-nodes that are now ocraft's memory organ.

This MCP closes that gap with a small tool set:
  list_jobs · run_task · service_status · tail_execution_log
  search_nodes · create_node · append_note
so Claude can DO things in ocraft, not just rewrite its code.

TWO framings, both true and both the point:
  1. It's the "REMEMBER" organ. Now that the knowledge base lives as html nodes,
     searching a note and appending to one ARE node ops. search_nodes/append_note
     make the node tree the durable scratchpad every other MCP's output needs a home
     in ("save this email summary as a note", "what did I write about X?").
  2. The REAL driver is the HEADLESS side. An interactive Claude Code session
     borrows Read/Grep/Edit/Bash from the harness. The scheduler and a chat-node
     "brain" have no harness — they run unattended. For them, "list my jobs / run
     this task / find that note / write this down" must arrive as MCP TOOLS, because
     there's no interactive toolbelt to borrow. This MCP is how the headless brain
     touches its own body.

## Fit with the node model / architecture
-------------------------------------------------------------------------------
The store is the source of truth (kernel/api.js). The automation engine
(runtime/) already exposes clean module APIs:
  - runtime/serviceManager.js: listServices, getService, start/stop/restart,
    readLog — a ready-made service surface.
  - runtime/taskExecutor.js: execute(name, args) — run a task to completion.
  - runtime/storage.js: listExecutions, loadTaskState, loadSchedulerState.
  - runtime/scheduler.js: the `jobs[]` array (currently a module const — see Open Qs
    re: list_jobs).
  - kernel/api.js: GET /api/nodes, GET /api/nodes/:id/body, POST /api/nodes,
    POST /api/nodes/:id/body — node CRUD + bodies.
The self-MCP is a THIN adapter over these, in the exact "house style" of
digitalocean-mcp: a shared *-api.js with the logic, a server.js with thin tool
shapers, importable by runtime tasks too.

It is "self" because it is a client of ocraft's own surfaces — same status as the
editor and the scheduler. It adds no new truth; it operates the existing one.

## How it reaches the store — via /api/* vs. direct (the core decision)
-------------------------------------------------------------------------------
Two ways for the MCP to touch nodes/services, and they trade off differently per
tool:

  A. OVER HTTP (fetch the local /api/*, :3001). Pros: one code path, the same one
     the editor uses; respects whatever the API enforces; works even if the MCP runs
     elsewhere. Cons: requires the `api` service to be up; node search would mean
     GET /api/nodes (whole list) + filter client-side (fine at this scale).
  B. DIRECT module import (call serviceManager.js / taskExecutor.js / storage.js,
     read node folders straight off disk). Pros: no running server needed — works
     headless even if `api` is down; richer access to executions/state that have no
     HTTP route today. Cons: couples the MCP to runtime internals; must replicate
     the api.js path-safety (the nodeDirSafe / ASSETS traversal guards).

RECOMMENDED SPLIT (pragmatic, not dogmatic):
  - NODES (search_nodes, create_node, append_note) -> HTTP /api/*. Node CRUD already
    has a clean, guarded HTTP surface (incl. the `..`-escape guard in nodeDirSafe and
    the server-minted-id POST). Reuse it; don't re-implement traversal safety.
  - AUTOMATION (list_jobs, run_task, service_status, tail_execution_log) -> DIRECT
    import of runtime/* . These have NO HTTP routes today, and the headless brain
    must work whether or not the `api`/editor is running. Importing
    serviceManager/taskExecutor/storage is the path of least new surface.
  This mirrors the DO MCP's shared-helper idea: a runtime/self/ shared module both
  server.js and runtime tasks import.

CONFIRM: the split above, vs. "all HTTP" (simplest, but needs `api` up and adds new
HTTP routes for jobs/executions) vs. "all direct" (most headless-robust, but the MCP
reaches into the store's guts and must own path-safety for node writes).

## Transport — stdio vs HTTP
-------------------------------------------------------------------------------
STDIO (like all four existing MCPs; registered in .mcp.json as `node server.js`).
  + Identical to the family; zero new infra; the interactive Claude session and the
    Agent SDK both speak stdio MCP. This is the default and the MVP.
  - A stdio server is spawned per-consumer and can't be supervised as a long-running
    service (the roadmap's process-supervisor item already notes this caveat).
HTTP/SSE transport.
  + Would let the headless brain (a long-lived chat-node process) and the scheduler
    share ONE running MCP instance, and bring it under serviceManager like any
    service. This matters specifically for the headless driver framing above.
  - More moving parts; needs the auth gate below; premature before the headless
    brain actually exists.
RECOMMENDATION: ship STDIO now (matches the family, unblocks interactive + Agent-SDK
use immediately). Note HTTP/SSE as the upgrade for when the always-on headless brain
lands and wants a shared, supervised instance — at which point it pairs naturally
with the ws-exchange `cmd.*` path (a chat-node could even reach these ops over the
hub). CONFIRM transport.

## Auth
-------------------------------------------------------------------------------
Unlike the other MCPs, this one needs NO external credential — it operates the local
machine. But that's exactly why it's DANGEROUS: run_task can execute any task
(do-droplet-up spends money; deploy git-pulls),
create_node mutates the store. Guard rails:
  - STDIO MVP: trust boundary = "who can spawn this process" (same as the other
    MCPs, same as the bypassPermissions ai-chat endpoint). It is LOCAL ONLY. Like
    ai-chat, it must NEVER be network-exposed. Document this loudly in the README,
    mirroring the kernel/api.js ai-chat warning.
  - HTTP/SSE (if/when): a shared-secret bearer, same spirit as WS_EXCHANGE_TOKEN.
  - Tool-level: make destructive tools explicit and narrow. run_task should take a
    task NAME + args and nothing that lets it run arbitrary shell; the *set* of tasks
    is the allowlist. Consider a `dangerous:true` flag in the tool description for
    money/destructive tasks (do-droplet-up, deploy) so the model is
    warned, the same way the DO delete tools shout "IRREVERSIBLE".
  - append_note must APPEND (read body, concat, POST body) — never clobber. Distinct
    from create_node so there's no boolean-flag overwrite footgun (matches the repo's
    no-boolean-flag-params rule).

## The tools (mirror the DO MCP's tool shape)
-------------------------------------------------------------------------------
  list_jobs            -> the scheduler's jobs[] (id, task, schedule/interval, last
                          run from scheduler state). READ.
  run_task             -> taskExecutor.execute(name, args); returns the execution
                          id + status + final logs. WRITE / can be dangerous.
  service_status       -> serviceManager.listServices() / getService(id). READ.
  tail_execution_log   -> storage.listExecutions() + read one execution's logs
                          (the executions/*.json records). READ.
  search_nodes         -> GET /api/nodes, filter by name/type/parent + body-text
                          (fetch /body for html nodes when text search is requested).
                          READ.
  create_node          -> POST /api/nodes (mint id) + optionally POST /body.
                          WRITE.
  append_note          -> resolve an html node, GET body, append text, POST body.
                          WRITE, append-only.
  (later) service_action(start|stop|restart, id), tail_service_log — pair with the
  process-management roadmap items; keep out of the MVP to stay tight.

server.js: thin ok()/fail() shapers + requireXxx() fail-fast in main(), exactly like
digitalocean-mcp/server.js. Logic lives in runtime/self/*.js so runtime tasks can
import the same ops (e.g. a "nightly digest" task that append_notes its summary).

## Open questions / risks
-------------------------------------------------------------------------------
  - jobs[] is a module-const in scheduler.js, not data. list_jobs needs it readable
    without running the loop — export the array (cheapest), or move jobs to a config
    file / nodes later (bigger). Recommend: export the const now.
  - run_task is a loaded gun: it can spend money / be destructive. The allowlist IS
    the task set, but the model could still fire do-droplet-up. Mitigate with loud
    "dangerous" descriptions + (optionally) excluding the worst from run_task and
    giving them dedicated, clearly-labelled tools. CONFIRM the danger policy.
  - direct-import path duplicates trust: writing nodes off-HTTP would bypass api.js's
    path guards — which is the whole reason node writes go through /api/* here, not
    direct. Keep that line.
  - executions are capped at 500 and rotate (storage.js) — tail_execution_log must
    handle a missing/rotated record gracefully.
  - overlap with the ai-chat endpoint: that endpoint already gives an *interactive*
    agent Read/Edit/Bash; this MCP is the *programmatic/headless* equivalent for
    OPERATING (not editing) the system. Don't merge them — different trust + lifecycle.

## Music note (parked, per the roadmap)
-------------------------------------------------------------------------------
A dedicated Music MCP is probably NOT worth building. For playback control the
claude-in-chrome extension already drives the Spotify/Apple Music WEB player, so the
"hands on playback" need is covered without an MCP. An API MCP would only pay off for
the DATA side (listening history, building playlists as node data) — a weaker, hazier
need. PARK it; revisit only if the history/logging-as-nodes use case becomes concrete
(at which point it'd reuse this self-MCP's create_node/append_note to land the data
as nodes anyway).

## Monetization
-------------------------------------------------------------------------------
No standalone angle — it's internal plumbing for a single user, not a sellable unit.
Honest indirect value: it's a load-bearing PREREQUISITE for the "personal life-
management agent" that a hosted ocraft could one day charge for — the headless brain
is worthless without hands on its own scheduler/notes, and this MCP is those hands.
So: don't charge for it; recognize it as the enabling layer under any future paid
"managed personal agent" story. Cheapest charge IF it ever ships hosted: it folds
into that agent subscription, never billed alone.
