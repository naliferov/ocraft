## What & why

Node.js landed a native QUIC + HTTP/3 implementation in **v26.2.0** via the
`node:quic` module — HTTP/3 over `nghttp3`, activated when the negotiated ALPN is
`h3`. It is **experimental** (Stability 1.0) behind the `--experimental-quic`
runtime flag. The idea: serve `kernel/api.js`'s `/api/*` over HTTP/3.

Be honest up front about the value, because it changes how this should be scoped:

  - `kernel/api.js` is a **localhost dev server** (port 3001, reached by Vite's
    proxy on the same machine). HTTP/3's real wins — no head-of-line blocking under
    packet loss, and connection migration across changing networks (wifi→cellular) —
    **do not apply on loopback**, where there is no loss and no network change.
  - QUIC **mandates TLS 1.3** and runs over **UDP**, whereas the API is plain HTTP
    over TCP today. So adopting it means standing up certs + a UDP listener for a
    server that currently needs neither.
  - The environment here is Node v22 (see `node --version`); `node:quic` needs
    v26.2.0+. So this can't even run on the current runtime without a Node upgrade.

So this is a **learning exercise**, not a performance need — and the plan treats it
that way. It is worth doing for what it teaches (QUIC handshake, ALPN, TLS 1.3 in
Node, the HTTP/3 framing) and for the demoable novelty, NOT because the editor will
load faster. The pragmatic production path (if HTTP/3 ever mattered for ocraft) is
the opposite of "rewrite the origin": terminate HTTP/3 at the edge and speak HTTP/2
to an unchanged Node origin (see "Production path" below).

## Fit with the node model / architecture

- This touches ONLY the transport in front of `kernel/api.js`'s `routes` table; it
  does not touch the node model, the store, or any node type. The route table
  (`'METHOD /path'` keys, `:param` matching, `sendText` gzip) is transport-agnostic
  and would be reused verbatim behind whichever listener answers.
- The architecturally clean shape is a thin **dispatch core** factored out of
  `kernel/api.js`: `handle(method, urlPath, req-like, res-like) -> route + params`,
  with the existing `http.createServer` callback and a NEW HTTP/3 handler both
  calling it. That keeps one source of truth for routes and lets HTTP/3 be an
  alternate front door rather than a fork of the server. (Today the request handler
  and routing are already separable — `matchUrl` + the `routes` object are pure; the
  `createServer` callback is the only http-coupled part.)
- Keep it OPT-IN and additive: a flag/env (e.g. `OCRAFT_HTTP3=1`) that starts the
  QUIC listener IN ADDITION to (or instead of) the plain HTTP one — never replace
  the working HTTP path, since the whole stack (Vite proxy, every `fetch('/api…')`
  in the frontend) depends on it and browsers won't speak h3 to it without Alt-Svc
  anyway (below).

## The honest blockers (name them, don't gloss)

  1. NODE VERSION. Needs v26.2.0+ for `node:quic`; this machine is v22, and the
     deploy box installs whatever NodeSource gives. Either bump the dev Node to 26
     for this experiment (and accept "experimental, behind a flag"), or run the
     experiment in a throwaway script under a v26 install. CONFIRM how you want to
     get a v26 runtime (nvm install 26 locally is simplest).
  2. EXPERIMENTAL + FLAG. `--experimental-quic` must be passed; the API surface can
     change between Node releases. Fine for a learning spike; do NOT make the
     default editor path depend on it.
  3. TLS 1.3 CERTS. QUIC won't run unencrypted. For localhost that means a
     self-signed cert (or a local CA via `mkcert`). A self-signed cert means clients
     must trust it or skip verification — manageable for a dev experiment, but it's
     real setup the plain-HTTP server never needed.
  4. UDP, not TCP. The QUIC listener binds UDP. Coexisting with the TCP HTTP server
     on the "same" port is fine (different protocols), but firewalls/Docker port
     mapping treat UDP separately — note it if this ever runs anywhere but the
     laptop.
  5. THE BROWSER WON'T JUST USE IT. A browser does not dial HTTP/3 directly; it
     learns a server supports h3 via an **`Alt-Svc`** response header on an
     HTTP/1.1/2 response, then *upgrades* subsequent requests to h3. And the
     immediate client here is **Vite's dev proxy**, whose http-proxy does not speak
     HTTP/3 to the upstream at all. So "the editor talks to the API over HTTP/3"
     is NOT achievable just by switching the origin — the proxy and the Alt-Svc
     advertisement are in the way. This is the single most important honesty point:
     the realistic deliverable is a STANDALONE HTTP/3 demo (a Node or curl client
     hitting the h3 listener directly), not the editor transparently using h3.

## Implementation plan (phased) — scoped as a spike

  Phase 1 — Refactor for reuse (no behavior change, valuable regardless). Extract a
    pure dispatcher from `kernel/api.js`: export `matchUrl` + a `dispatch(method,
    urlPath)` that returns `{handler, params}`, and keep the `http.createServer`
    callback as the thin TCP adapter that calls it. This is the only change that
    touches the working server, and it's a safe, useful decoupling on its own.
  Phase 2 — Certs. Generate a localhost TLS 1.3 cert (mkcert for a trusted local CA,
    or `openssl`/`selfsigned` for a self-signed one). Store under a gitignored dev
    path; never commit keys.
  Phase 3 — HTTP/3 listener (the actual learning). A separate `kernel/api-http3.js`
    (run under Node 26 with `--experimental-quic`) that: creates a `node:quic`
    server with the cert, ALPN `['h3']`, wires the HTTP/3 session/stream events to
    build a minimal `{method, path, headers, body}` and a minimal response shim, and
    calls the SAME `dispatch()` from Phase 1, then writes the route's result back
    over the h3 stream. Reuse `sendText`'s gzip logic if practical. Goal: GET
    `/api/nodes` returns the node list over h3.
  Phase 4 — Prove it. A tiny client that speaks h3 (a Node `node:quic` client, or
    `curl --http3` if the local curl is built with HTTP/3) hitting the listener and
    printing the node JSON. THIS is the demo artifact. Capture the handshake/ALPN in
    logs.
  Phase 5 (optional, to "close the loop" toward the editor) — Advertise via
    `Alt-Svc`. Add an `Alt-Svc: h3=":3001"; ma=…` header to the plain HTTP server's
    responses so an h3-capable BROWSER (hitting the API directly, not through Vite)
    would upgrade. Document that Vite's proxy still won't; this is illustrative.

## Production path (the pragmatic note from the roadmap)

If HTTP/3 ever genuinely matters for ocraft (it doesn't for localhost), do NOT serve
h3 from the Node origin in production. Instead:

  - Terminate HTTP/3 **at the edge** — a CDN, or nginx 1.25+ / Caddy with QUIC — and
    speak **HTTP/2 (or 1.1) to the unchanged Node origin** over the loopback/private
    net. The edge handles certs, UDP, congestion control, and connection migration;
    the origin stays plain. The edge advertises h3 to clients via `Alt-Svc`.
  - This is exactly how HTTP/3 is deployed in practice and means the Node API never
    needs `node:quic` in production at all. The native module is interesting to
    *learn*; the edge is where you'd actually *ship* it.

So the two framings stay separate and honest: Phases 1–4 are a learning spike on
`node:quic`; the production answer is "edge terminates h3, origin stays HTTP/2."

## Open questions / risks

  - Node 26 availability: needs v26.2.0+ with `--experimental-quic`; dev box is v22.
    CONFIRM how the v26 runtime is obtained (nvm) — and that it's only for the spike,
    not the editor's default path.
  - `node:quic` API churn (Stability 1.0): the exact session/stream/HTTP-3 surface
    may differ from any blog example — verify against the v26.2.0 docs at build time.
  - Self-signed trust: clients must trust or skip verification; mkcert avoids the
    "skip verification" footgun by installing a local CA.
  - Vite proxy can't reach an h3 upstream → the editor won't transparently use h3;
    set expectations to a standalone demo, not an editor speedup.
  - Zero performance benefit on loopback — do not justify this as perf; justify it as
    learning + a novelty demo. If you want it to do something real, the edge
    termination path is the answer, and that needs no Node QUIC.
  - Don't regress the working HTTP path: keep h3 strictly additive/opt-in.

## Monetization

None, and it would be dishonest to invent one. This is a personal localhost dev
server; HTTP/3 changes nothing a user would pay for, and the real-world benefits
don't even apply on loopback. There is no paid tier, no metered usage, no template
pack here. The only value is educational (understanding QUIC/HTTP/3 in Node) and a
small content/novelty payoff (below). State plainly: no monetization angle.

## Marketing

Niche but genuinely shareable as a "TIL/curiosity" piece, because native HTTP/3 in
Node is new and most devs haven't tried it. Artifact: a short clip/post — `curl
--http3 https://localhost:3001/api/nodes` (or the tiny Node h3 client) returning the
node JSON, with the QUIC handshake/ALPN=`h3` visible in the server log, captioned
"serving my app's API over HTTP/3 with Node's native `node:quic` — zero proxies."
Hook: "Node has native HTTP/3 now (v26.2, experimental) — here's the smallest
possible server." Channel: X / Hacker News / r/node — this is a developer-curiosity
audience, not the ocraft node-OS audience, so frame it as a standalone Node TIL, not
an ocraft feature. Target: backend devs tracking Node releases. Pair it with an
honest "...and here's why you'd actually terminate it at the edge, not the origin"
follow-up — the honesty is itself good content. Lowest priority of the marketing
artifacts; do it because it's fun, not for reach.
