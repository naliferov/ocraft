 `yo` as a node scripting medium
 (roadmap: "`yo` as a node scripting medium")
===============================================================================

## What & why

`yo` is a tiny, async-first toy language whose keywords are `@`-mentions
("yo, @log this"): `name = expr` bindings, `@a`/`@f` async/sync functions,
`@w` await, `@r` return, `@l` print, arithmetic. It now lives as a **script
node** — `yo-to-js` (id 200, under `scripts/`, body `data/nodes/200/script.js`)
— that **transpiles `yo` to JavaScript** and runs the emitted JS in the browser.

STRATEGY CHANGE (was: interpret → now: compile). The old
`experiments/js-engine/yo.js` was a full lexer → parser → **evaluator**: it
walked the AST and produced values directly. That file is deleted. The script
node keeps the same lexer + parser but replaces the evaluator with a **code
generator** that emits JS source, which a `new Function(...)` then runs. The
mapping is almost 1:1 because yo's async model IS JS's:

    @a -> …    →   async () => …      (a call returns a Promise)
    @f -> …    →   () => …            (a call returns the value)
    @w e       →   (await e)
    @r e       →   return e
    @l         →   log                (the print builtin, injected at run)
    name = e   →   let name = e       (first bind per scope) / name = e (rebind)

Compile shows the generated JS; Run executes it. The whole pipeline
(tokenize → parse → compileToJs) is plain, runtime-agnostic JS.

Why transpile instead of interpret: (1) **simplicity** — a code generator is
smaller and clearer than an async tree-walking evaluator with `Async`/
`ReturnSignal` plumbing; (2) it produces a **real, inspectable artifact** (you
see the JS), which is a better teaching/demo story and a better "a note that is
a program" story; (3) it runs at JS speed with no per-node `evaluate` overhead.

## Honest caveat — transpiling FORFEITS the old sandbox argument

The previous plan's headline justification for `yo` was *capability-by-
construction*: an interpreter only lets a guest do what its `evaluate` switch
allows, so a `yo` node's powers were exactly the builtins we seeded (`@l`,
`@x`, …) — an allow-list instead of "everything the browser can do." **Compiling
to JS and `new Function`-ing it throws that away**: the emitted code is ordinary
JavaScript with full page access. So the transpile path is NOT a safety story.

That leaves a real fork to decide before investing further:
  - If the goal is **a language that runs + pedagogy + a clean demo**, transpile
    is the right call (where we are now), and safety comes from the *general*
    sandbox — wrap the generated JS in a Worker/iframe exactly like any other
    `script` node (see sandboxed-execution-plan.txt). `yo` gets no special
    isolation; it's just JS by the time it runs.
  - If the goal is **capability-scoped untrusted execution** (the multi-user
    "users run `yo`, not raw JS" story), transpile-to-JS does not deliver it —
    that wants either the interpreter back, or transpiling to a *constrained*
    target (a restricted runtime / QuickJS-in-WASM). Don't claim the safety
    benefit while emitting raw JS.

## Fit with the node model / architecture

It's a `script` node, so it already fits — no new node type, no `NODE_BODY`
entry, no `Yo.vue` handler. The transpiler reuses the script-node surfaces: the
`x.ui` panel (source textarea + generated-JS pane + log) and, if it grows
composition, `x.x`/`x.log` directly. Because the compiler is just code in a
node body, it's authored, versioned, and composed like everything else — and a
second node could call it (`x.x("200", { src })`) to get JS back, making "yo →
js" a reusable build step rather than a one-off demo.

## Remaining work (language maturation — now CODEGEN cases, not evaluator cases)

`yo` is expressive enough to demo (`onePlusOne` sample) but too thin to script
real nodes. Each gap below is now "add a `genExpr`/`genStmt` case + a lexer/
parser case," in rough priority:

  - **Function parameters.** `Fn` ignores call args today (`@a -> …` compiles to
    `() => …`). Parse a param list (`@a (a, b) -> …`) and emit `(a, b) => …`;
    bind call args positionally. Highest priority — without it `@x`-style
    composition can't pass values.
  - **Strings.** The lexer has no string literal (numbers/idents only). Add
    `"…"` → a `String` node → emit a JS string literal (JSON.stringify the
    value so escaping is correct).
  - **Collections + member access.** lists/objects and `a.b` / `a[i]`, so a `yo`
    program can handle the JSON shapes `x.x` returns. Direct JS emission.
  - **Comparison + branching.** `==`/`<`/`>`, `@if`/`@else` (or a ternary),
    booleans — `BinaryOp` codegen currently emits only `+ - * /`.
  - **Iteration.** A bounded loop (`@each`/`@times`) — bounded on purpose.
  - **Error ergonomics.** Carry line/column from the lexer through parse so a
    compile error reports `error at line N`, not a bare message.
  - **Composition builtin.** project `@x(id, args)` → `x.x` (cycle guard already
    in `makeCtx`) and `@log` → `x.log`, injected as `new Function` params next to
    `log`, so a `yo` program can call other nodes.

## Headless reuse (free, because codegen is runtime-agnostic)

The same `tokenize → parse → compileToJs` runs in Node. A scheduler task could
load a `yo` body, compile it, and run the JS with a backend builtins projection
(`@x` → headless node runner, `@log` → execution logs) — a portable automation
unit. Note this inherits the caveat above: headless `new Function` is full Node
access, so it needs the backend sandbox before running untrusted `yo`.

## Key files / components
  - `data/nodes/200/script.js` — the `yo-to-js` script node: lexer + parser +
    `compileToJs` + the `x.ui` panel. The home of the language now.
  - (gone) `experiments/js-engine/yo.js` + `program.yo` — the old interpreter,
    deleted; the sample program lives inline in node 200 as `SAMPLE`.
  - sandboxed-execution-plan.txt — where real isolation for the *generated JS*
    comes from (Worker/iframe), if/when untrusted `yo` matters.

## Open questions / risks
  - The fork above (demo-language vs. capability-scoped execution) is THE
    decision — it determines whether transpile-to-JS is the end state or a
    stepping stone back toward a constrained target. Decide before Phase-3
    breadth.
  - Build-vs-buy unchanged: a hand-rolled toy will trail QuickJS-in-WASM on
    power AND (now) on safety. `yo`'s justification is pedagogy + the "a note
    that is a program" demo, not power or sandboxing.
  - Scope creep: each maturation item is "add a real language feature." Gate on a
    concrete target program (e.g. "a `yo` node that polls another node and logs a
    digest") and build only what it needs.

## Monetization
No direct angle, and that's fine — say so plainly. `yo` is differentiation /
pedagogy, not a sellable unit; nobody pays for a bespoke toy language. (Note the
old indirect pitch — "the safe execution story for multi-user" — is WEAKER now
that we emit raw JS; don't lean on it.) If anything ever charges, it's the
hosted platform, not `yo`.

## Marketing
Strong "show, don't tell" / educational artifact. The transpile angle is arguably
a BETTER build-along than the interpreter was: "I wrote a tiny language and made
it compile to JavaScript, live, inside my node app" — show source → tokens →
AST → **generated JS** → it runs. Fits `plans/youtube-content-plan.txt`: visual
(watch the JS appear), teaches a beloved topic (write a compiler), demonstrates
the node-OS differentiator. Hook: "`yo, @log this` → here's the JavaScript it
becomes." Channel: YouTube long-form + an X clip of the source/JS split panel.
Target: language-nerd / compiler-curious devs.
