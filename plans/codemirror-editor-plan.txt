## What & why
The script editor today is a bare <textarea> (frontend/src/components/NodeItem/
nodeTypes/Script.vue:152) — no syntax highlighting, no line numbers, no bracket
matching, hand-rolled Tab-inserts-two-spaces, and the only smartness is a regex
that injects `/*→name*/` hints after `x.x("id")`. The README's "a code editor for
scripts" is aspirational. Replace the textarea with CodeMirror 6:

  Phase A (table stakes): JS highlighting + line numbers + bracket matching +
    auto-indent + a dark theme matching the current #1a1a1a / #d4d4d4 look.
  Phase B (the load-bearing part): autocomplete on the `x` API — `x.x()`, `x.ui.*`,
    `x.auth.*`, `x.args`, `x.log` — via a CM6 completion source. THIS is what shows
    off "a note that *is* a program"; generic highlighting is just polish.

Editor choice (README already argues it; recap honestly):
  - Monaco — VS Code's editor, real IntelliSense, but multi-MB + fiddly Vite worker
    config. The frontend bundle is already ~1.5 MB (Three.js, p5, Tone, naive-ui);
    Monaco would dominate it. Overkill for a single-user local tool.
  - CodeJar + Prism — ultra-minimal (highlight-only, contenteditable). Cheapest, but
    no real autocomplete API, no gutter, weak as a long-term base.
  - CodeMirror 6 — tens of KB, tree-shakeable, no web-worker setup, first-class
    `autocompletion({ override: [...] })` extension point. Wins for this tool.
  Decision: CM6. The same EditorView config can later back the HTML source view and
  a future `yo` language mode (see §Open questions).

## Fit with the node model / architecture
Pure frontend; touches no kernel code, no API, no node storage. Script bodies are
still fetched/saved via GET/POST /api/nodes/:id/body exactly as now — CM6 only
changes how the buffer is *displayed and edited*. The component still exposes
save() to NodeItem.vue's single Save, still strips/injects the `/*→name*/` marks,
still owns the x.ui host above the editor. The `x`-API completion source is the
editor-side mirror of the runtime `x` shape defined in useNodeScripts.js
(makeCtx) + useScriptUi.js (the ui object) — so completion keeps a small, honest
contract with what actually runs.

## Implementation plan (phased) — concrete steps, contracts, real files

PHASE A — swap textarea -> CM6 (highlighting, gutter, brackets)
  1. Deps (frontend/): `npm i codemirror @codemirror/lang-javascript`. The
     `codemirror` meta-package re-exports the core bundle (state, view, commands,
     language, search, autocomplete) — enough for Phase A+B without hand-picking
     every @codemirror/* sub-package. Confirm tree-shaking keeps it ~tens of KB in
     the built chunk (check `vite build` output; Script.vue is not lazy today, so
     this lands in the main bundle — acceptable, but note it).
  2. New composable: frontend/src/composables/useCodeEditor.js — owns the CM6
     EditorView lifecycle so Script.vue stays declarative and a second host (HTML
     source, yo) can reuse it. Shape:
        createCodeEditor({ parent, doc, onChange, extensions = [] }) -> {
          view, getDoc(), setDoc(text), focus(),
          insertAtCursor(snippet),   // replaces the manual selectionStart splice
          destroy()
        }
     Base extensions: lineNumbers(), highlightActiveLine(), bracketMatching(),
     closeBrackets(), history(), indentOnInput(), javascript(), the dark theme,
     keymap.of([...defaultKeymap, ...historyKeymap, ...closeBracketsKeymap,
     indentWithTab]). `indentWithTab` replaces the bespoke onKeydown Tab handler.
     onChange fires an EditorView updateListener -> Script.vue's scriptCode ref
     (keep the ref as the source of truth so the dirty-check / strip-on-save logic
     is untouched).
  3. Theme: an EditorView.theme({...}) port of the current .editor CSS — background
     #1a1a1a, color #d4d4d4, font JetBrains Mono 13px/1.6, padding 12px, border
     #333, radius 4px — plus a HighlightStyle (or @codemirror/theme-one-dark, but a
     hand-tuned ~20-line theme avoids the extra dep and matches the app palette).
  4. Script.vue surgery:
       - Drop <textarea>, the txtAreaRef, onKeydown (Tab handling moves to CM6),
         and the manual selectionStart/Range splices in insertCall.
       - Add <div ref="editorHostRef" class="editor" />; mount the editor in
         onMounted *after* the body fetch resolves (set initial doc to the
         annotated source). Tear down in onBeforeUnmount (view.destroy()).
       - scriptCode stays a ref but is now driven by the onChange callback; save()
         reads view.getDoc() (or the mirrored ref) — strip-marks logic unchanged.
       - insertCall(id) -> editor.insertAtCursor(snippet) + focus; the snippet
         (`x.x("id" /*→name*/)`) and the marks stay exactly as today.
       - annotate()/stripMarks()/MARK regex: KEEP AS-IS. They operate on the text
         string, independent of the widget. (Future nicety: render the hint as a CM6
         atomic decoration widget instead of inline comment text — out of scope;
         note it.)
  5. Verify: open node 11 (websocket-tester, runOnOpen) and a wasm node (10/4);
     highlighting + gutter show, Tab indents, Run still works, Save round-trips,
     the x.ui panel still mounts above the editor, the `/*→name*/` hints still
     appear on load and are stripped on save.

PHASE B — `x`-API autocomplete (the differentiator)
  6. New module: frontend/src/composables/useScriptCompletions.js — exports a CM6
     CompletionSource `xApiCompletions(context)`. It is the single declarative
     description of the `x` surface, kept next to (and honestly in sync with) the
     runtime in useNodeScripts.js / useScriptUi.js:
       - Trigger after `x.` -> top-level members: x(), ui, auth, args, log.
       - After `x.ui.` -> row, input, button, text, log, onCleanup (from
         createScriptUi's `ui` object).
       - After `x.auth.` -> get, set, clear (from authStore).
       - `x.x(` -> a completion list of callable script-node ids, label
         "name · id", inserting `"id"` — REUSE store.nodes filtered to type==='script'
         && id !== self, exactly the callableOptions computed already in Script.vue.
         This folds the "Insert call" popselect into inline completion (keep the
         button too — discoverable for new users).
     Each option carries `type` (CM6 icon: 'method'/'property'/'variable'),
     `detail` (signature, e.g. `x.x(id, args?) => Promise`), and `info` (one-line
     doc from the README/makeCtx comments). Use context.matchBefore(/x(\.\w*)*$/)
     to scope; return { from, options }.
  7. Wire it: pass `autocompletion({ override: [xApiCompletions] })` (plus the
     default JS-keyword completion via languageData) into the editor extensions in
     useCodeEditor. Bind Ctrl-Space (startCompletion) in the keymap.
  8. (Optional, nice) A tiny hover/tooltip via hoverTooltip showing the `x.x("id")`
     target node's name — supersedes the inline `/*→name*/` marks eventually, but
     don't remove the marks in this pass (they also annotate the SAVED-then-reloaded
     source path and are zero-dep).
  9. Verify: typing `x.` and `x.ui.` and `x.auth.` pops the right members;
     `x.x("` lists script nodes by name; picking inserts the bare id.

## Open questions / risks
  - Bundle: Script.vue is eagerly imported in NodeItem.vue. If CM6 measurably grows
    the main chunk, lazy-load Script.vue via defineAsyncComponent like Scene3d
    already is (NodeItem.vue:34). Decide after measuring.
  - Completion staleness: useScriptCompletions duplicates the `x` shape. Keep it a
    single small object literal so a drift is one-file-obvious; add a code comment
    cross-linking makeCtx. Not auto-derived (the runtime builds `x` imperatively).
  - The `/*→name*/` inline marks vs. a CM6 decoration widget — leaving marks as text
    is simplest; revisit only if they get in the way of completion.
  - WASM (AssemblyScript) nodes share this editor but are not JavaScript. Acceptable:
    JS highlighting is a close superset for AS; a dedicated AS mode is out of scope.

## Monetization
No standalone angle — this is table-stakes polish, not a paid feature. Don't charge
for it. It only matters as a baseline *if* ocraft ever became a hosted, multi-user
"scriptable node OS," where a decent editor is the price of entry, not a line item.
Honest answer: no realistic direct monetization; it raises the perceived quality
that makes everything else sellable.

## Marketing
Highly demoable. Weak artifact: a before/after clip (raw textarea -> highlighted,
line-numbered editor). KILLER artifact: a screen-recording of `x.`-API autocomplete
— typing `x.x("` and watching real node names pop, then `x.ui.button(...)` building
a live panel above the editor in one take. That shows the "a note that *is* a
program" thesis far better than generic highlighting. Channel: a short clip on
X / a YouTube short (fits plans/youtube-content-plan.txt), hook: "my notes app has
autocomplete for calling other notes." Target user: developers who like the
notes-are-programs idea.

## Effort — S/M/L
  M overall.
  - Phase A (textarea -> CM6, theme, composable, rewire insertCall/save): S–M,
    most of the risk is matching the current look + keeping the marks/dirty-check
    logic intact.
  - Phase B (x-API completion source + wiring): S, but it's the high-value half —
    do not skip it; it's the reason to do this at all.
  - Bundle check + optional lazy-load of Script.vue: XS.
