 PIXEL-ART SVG ASSETS FOR SCENES — plan
 (roadmap: "Pixel-art SVG assets for scenes")
 (written 2026-06-19; for later review — decisions marked "CONFIRM" are yours)

## What & why

Bring a set of **pixel-art sprites/clipart as SVGs** that `scene2d` nodes drop
in as elements, extending the clipart-SVG pipeline that already exists (assets
in `kernel/data/assets/img/svg/`, served read-only via `/api/assets/<relpath>`,
loaded and blitted by the p5 renderer). Pixel art authored AS SVG (a grid of
flat-fill `<rect>` cells, or a small viewBox with crisp paths) is:

  - crisp at any zoom — the scene letterboxes/scales the viewport up
    arbitrarily, and vectors don't soften the way a raster sprite would;
  - tiny and diff-able in the repo — a sprite is a few KB of text, reviewable in
    a PR, vs. an opaque binary sprite sheet;
  - composable as scene ELEMENTS — it slots straight into the existing
    `type: 'clipart'` element with no new rendering path;
  - a natural medium that pairs with the generative-art direction
    (`plans/generative-art-plan.txt`) — pixel-art scenes are recordable demos.

The pipeline already works: nodes 1 ("car in forest") and 25 reference
`tree.svg`/`car.svg` clipart today. What's MISSING and what this plan adds:
  (1) there's no way to ADD/insert a clipart element from the editor — elements
      are hand-edited into `state.json`; the only in-editor interaction is
      drag-to-move an element that already exists (`pickElement` in
      `renderSceneP5.js`). A small picker/library closes this.
  (2) there are only 4 SVGs today (car, fish, tree, tree2) and they're SMOOTH
      flat vector art, not pixel art — we need an actual pixel-art SET.
  (3) ONE load-bearing rendering detail (see "Open questions"): the renderer
      rasterizes each SVG to an offscreen canvas with DEFAULT smoothing ON, then
      blits it. That blurs true pixel art on upscale — pixel art needs
      smoothing OFF on both the offscreen rasterize AND the per-frame blit.

This is a small, self-contained, demoable feature that reuses almost everything.


## Fit with the node model / architecture

It rides the EXISTING clipart element model — no new node type, no new file
format, no API change for storage. A `scene2d` node's `state.json` holds an
`elements[]` array; a clipart element is exactly:

    {
      "name": "hero",
      "type": "clipart",
      "z": 2,
      "props": {
        "src": "pixel/hero.svg",   // bare name -> img/svg/<name>; a path with
                                    // '/' is relative to assets/, so a
                                    // "pixel/" subfolder works as-is today
        "x": 80, "y": 45,           // CENTER, in logical viewport units
        "width": 16, "height": 16,  // logical size (one may be omitted -> derived
                                    //   from the SVG aspect; both omitted -> ~40w)
        "scale": 1,
        "rotation": 0               // degrees, clockwise
      },
      "state": {},
      "events": []
    }

(Verified against node 1's existing clipart elements and the `drawClipart` /
`assetUrl` code in `renderSceneP5.js`.)

How the renderer handles it today (so the plan stays accurate):
  - `assetUrl(src)`: a bare name resolves to `img/svg/<name>`; a src containing
    '/' is taken relative to `assets/`. So pixel-art sprites can live in a
    `kernel/data/assets/img/svg/pixel/` subfolder and be referenced as
    `pixel/hero.svg` with ZERO renderer change.
  - `getClipart(src)`: loads each SVG ONCE, rasterizes it to a high-res
    offscreen canvas (`RASTER_LONG_SIDE = 1024` on the longest side), caches it,
    and blits that bitmap every frame. Good for performance; the smoothing flag
    is the only thing to fix for crisp pixels (below).
  - Layering by `element.z`; drag-to-move + hit-test by `pickElement` /
    `elementBounds` already support clipart. So a placed pixel-art sprite is
    immediately draggable and z-orderable — nothing extra needed there.

The ONLY new surfaces are: (a) a sprite library/picker UI in the scene editor,
(b) the smoothing fix for crisp upscaling, (c) the actual sprite SET. Storage,
serving, layering, and dragging are all already done.

CONVENTIONS reminder: plain `.js` ESM; develop on `main`; no single-letter vars;
no boolean flag params (e.g. a render mode is a named string `'smooth'`/
`'pixel'`, not a boolean — see the smoothing decision below).


## Implementation plan (phased)

PHASE 1 — make pixel art render CRISP (prerequisite; tiny, highest-signal).
  - In `getClipart`'s offscreen rasterize (`off.getContext('2d')`) and in
    `drawClipart`'s per-frame `sketch.drawingContext.drawImage`, disable image
    smoothing so upscaled pixels stay hard-edged.
  - Decide the SCOPE of "no smoothing": per-element opt-in vs. global. Smooth
    clipart (the existing car/tree) WANTS smoothing; pixel art does NOT. So this
    must be a per-element choice, NOT a global flag. Add an element prop —
    e.g. `props.render: 'pixel' | 'smooth'` (default 'smooth' to preserve
    current behavior) — and have the renderer set `imageSmoothingEnabled`
    accordingly on both the offscreen context and the blit. A NAMED mode string,
    not a boolean param (per conventions). The offscreen canvas should also be
    sized to a small integer multiple of the sprite's native pixel grid for true
    nearest-neighbor crispness, rather than always 1024 — CONFIRM whether to
    special-case the raster size for `render: 'pixel'`.
  - Acceptance: a pixel sprite placed at width 16 in a 160-wide viewport, scaled
    up by the letterbox, shows hard pixel edges (no blur).

PHASE 2 — a sprite LIBRARY/PICKER in the scene editor.
  - Today `Scene2d.vue` has a transport + timeline + the p5 canvas, but NO
    add-element affordance. Add a small picker: list the available SVGs (from
    `img/svg/` and a `img/svg/pixel/` subfolder), show thumbnails, and on click
    APPEND a new clipart element to `node.elements` (centered, sensible default
    size, `render: 'pixel'` for pixel sprites). Save persists it (no autosave —
    same as every field, per the drag-to-move comment in Scene2d.vue).
  - Source the list from the assets dir. Cheapest: a tiny API addition —
    `GET /api/assets-list?dir=img/svg` returning filenames (the asset serving is
    already a read-only, traversal-guarded handler in `kernel/api.js`; a sibling
    listing route is small). Avoid hardcoding the filename list in the frontend.
    CONFIRM: API listing endpoint vs. a static manifest JSON shipped in assets.
  - This is also the natural home for an element panel generally (add/remove/
    reorder/toggle `enabled`) — but keep THIS phase to "pick a sprite -> it
    appears," matching ocraft's "smallest useful step" habit. A full element
    inspector is a follow-up.

PHASE 3 — the actual pixel-art SPRITE SET (the content).
  - Decide AUTHORING approach (the Roadmap's open question — "author by hand,
    generate, or import a set"):
      (a) HAND-AUTHOR as SVG `<rect>` grids — full control, on-brand with the
          existing hand-made car/tree SVGs, but slow per sprite.
      (b) GENERATE — write a tiny converter: a pixel grid (a 2D array of palette
          indices, or a small PNG) -> an SVG of merged flat-fill rects. This is
          the best fit for ocraft: the converter itself is a `script` node
          (`x.x`-callable), turning "draw/define a grid" into "emit a diffable
          SVG." Pairs perfectly with the generative-art track (a script could
          GENERATE pixel sprites procedurally). RECOMMENDED primary path.
      (c) IMPORT a set — pull a permissively-licensed pixel-art pack and convert
          raster sprites to SVG via (b)'s converter. Fastest to a populated
          library; must check LICENSING before committing anything to this
          public repo (mirror the importer's care about excluded content).
  - RECOMMENDATION: build (b) the grid->SVG converter first (small, reusable,
    on-brand), seed a handful of hand-defined sprites through it, and keep (c)
    as a way to bulk-fill later with license-clean art.
  - Store under `kernel/data/assets/img/svg/pixel/`. Keep each sprite small and
    diffable; flat fills only (no gradients/filters) so they stay crisp + tiny.

PHASE 4 (optional, later) — polish: per-element inspector (size/scale/rotation/
  z/enabled fields), a palette-swap prop (re-tint a sprite by remapping fills),
  and a tie-in to generative-art (procedurally generated sprites as scene
  elements).


## Key files / components

  - frontend/src/lib/renderSceneP5.js — the clipart pipeline: `assetUrl`,
    `getClipart` (offscreen rasterize, `RASTER_LONG_SIDE`), `drawClipart`
    (per-frame blit), and the `elements`/`pickElement`/`elementBounds` model.
    Phase-1 smoothing fix + reading `props.render` live here.
  - frontend/src/components/NodeItem/nodeTypes/Scene2d.vue — the scene editor;
    add the sprite picker/library UI here (it already owns the p5 canvas, drag
    logic, and the toolbar slot).
  - kernel/api.js — `serveAsset` + `ASSETS_PREFIX` (read-only, traversal-guarded
    asset serving). Phase-2 adds a sibling DIRECTORY-LISTING route for the
    picker.
  - kernel/data/assets/img/svg/  (+ new pixel/ subfolder) — where sprites live;
    referenced from `state.json` clipart elements as `pixel/<name>.svg`.
  - a new grid->SVG converter `script` node (Phase 3b) under
    kernel/data/nodes/<id>/script.js — pixel grid in, diffable SVG out;
    `x.x`-callable so it composes with generative-art scripts.
  - example: kernel/data/nodes/1/state.json — the existing clipart-using scene,
    the reference shape for a clipart element.


## Open questions / risks

  - SMOOTHING IS LOAD-BEARING (the one real gotcha): the renderer rasterizes to a
    1024px offscreen canvas AND blits per frame, both with default smoothing ON.
    Pixel art MUST set `imageSmoothingEnabled = false` on BOTH or it blurs on
    upscale. This is the make-or-break detail — verify a placed sprite is crisp
    before building the library on top. Per-element `render: 'pixel'`, not
    global (existing smooth clipart must keep smoothing).
  - RASTER SIZE vs. native grid: blindly rasterizing a 16x16 sprite to 1024px
    then nearest-neighbor-upscaling is fine, but rasterizing to an exact integer
    multiple of the native grid avoids any half-pixel seams. CONFIRM whether to
    special-case raster size for pixel sprites.
  - SVG flavor: "pixel art as SVG" can mean (i) a literal grid of `<rect>` cells,
    or (ii) crisp paths in a small viewBox. (i) is the truest, most diffable, and
    what the converter emits; standardize on it to keep sprites tiny and the
    palette obvious. Filters/gradients are banned (defeat crispness + bloat).
  - PICKER DATA SOURCE: live API directory listing vs. a checked-in manifest.
    Listing is more dynamic (drop a file -> it shows up); a manifest is simpler
    and cache-friendly. CONFIRM.
  - LICENSING for the import path (3c): never commit non-permissive art into this
    public repo — same discipline the README flags for the notes importer's
    EXCLUDED set.
  - SCOPE creep: a full element inspector is tempting but is Phase 4 — keep the
    first cut to "pick a sprite -> it lands -> drag it" (which the existing drag
    code already supports).


## Monetization (who'd pay, why, cheapest charge)

  - SPRITE / SCENE-ASSET PACKS — the direct angle, and it ties into the
    generative-art plan's same pack idea. Curated pixel-art SVG sets (characters,
    tiles, props, UI bits) sold as a small bundle (Gumroad/itch.io zip, or a
    paid template-pack drop). Who pays: indie game devs, no-code/creative-tool
    users, scene-makers who want ready art. Cheapest charge: a one-off paid pack
    download — no infra, no billing system needed beyond a storefront link.
  - The grid->SVG CONVERTER and the picker themselves: NO standalone sell —
    they're tooling that makes the packs (and the editor) better, not a product.
    Honest: the money is in the ART (packs), not the plumbing.
  - A hosted "sprite editor" SaaS: out of scope and not worth it for a
    single-user local tool; no near-term angle.
